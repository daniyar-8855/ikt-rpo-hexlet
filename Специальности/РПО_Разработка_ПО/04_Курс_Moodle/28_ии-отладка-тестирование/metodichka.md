## 1. Введение: зачем тебе эта тема

Представь обычный рабочий момент. Ты дописал функцию, нажимаешь «Запустить» — и вместо результата на пол-экрана вываливается красный текст с непонятными словами `Traceback`, `Error`, именами файлов и номерами строк. Первое желание новичка — закрыть глаза на этот текст и начать менять строки кода наугад: убрать одну скобку, поменять `+` на `-`, обернуть всё в `try`, «а вдруг заработает». Иногда «срабатывает» — программа перестаёт падать. Но баг при этом никуда не делся: он просто спрятался и вернётся позже, в самый неподходящий момент — на защите проекта или у реального пользователя.

Так работать нельзя, и это понимали задолго до появления ИИ. Есть профессиональный навык — **отладка (debugging)**: умение спокойно прочитать сообщение об ошибке, понять причину и устранить именно её, а не симптом. Сегодня у тебя появился сильный союзник: ИИ-ассистент умеет прочитать тот самый «страшный красный текст», объяснить его простыми словами, предложить вероятную причину и даже придумать тесты, на которых код сломается. Это резко ускоряет работу.

Но здесь же кроется ловушка. ИИ может предложить решение, которое *маскирует* баг, а не устраняет его. Он может сослаться на функцию, которой нет. Он может «починить» одно и сломать другое. Если ты принимаешь его ответ вслепую — ты не отладил код, а только переложил ответственность на инструмент, который никакой ответственности не несёт. Поэтому тема звучит именно так: отлаживать и тестировать код **вместе с ИИ, не теряя контроль**.

Почему это важно лично тебе? Потому что большую часть рабочего времени разработчик не пишет новый код «с нуля», а **ищет и исправляет ошибки** в уже написанном — своём и чужом — и закрывает их тестами, чтобы баг не вернулся. Тот, кто умеет быстро находить причину и фиксировать её тестом, экономит часы и не выпускает в продакшн поломанную программу. Это ежедневная работа, и она напрямую связана с твоей будущей специальностью и профессиональным модулем «Тестирование ПО» [1; 5].

После изучения темы ты сможешь:

## 3. Ключевые понятия

- **Отладка (debugging)** — поиск и устранение причины ошибки в коде [9].
- **Traceback (трассировка)** — сообщение, которое Python печатает при падении программы: тип ошибки, цепочка вызовов, файл, строка и описание [9].
- **Исключение (exception)** — ошибка, возникающая во время выполнения программы и прерывающая её, если не обработана [9].
- **Тест** — код, который автоматически проверяет, что твой код работает правильно [11].
- **Крайний случай (edge case)** — граничные данные, на которых код чаще всего ломается: пусто, ноль, отрицательное, очень большое значение.
- **Маскировка бага** — приём, при котором ошибка скрывается (например, «проглатывается» через `try/except`), а её причина остаётся.
- **assert** — оператор Python, который проверяет условие и роняет программу с `AssertionError`, если оно ложно [9].
- **pytest** — популярный инструмент для автоматического запуска тестов в Python [11].

## 4. Подробная теория

Это ядро темы. Сначала разберёмся, что такое ошибки и как читать traceback (без этого отладка невозможна). Затем пройдём весь алгоритм отладки по шагам с реальными примерами. После — научимся правильно и безопасно подключать ИИ. И в финале перейдём к тестированию: три вида тестов, генерация тестов с ИИ и главная мысль темы — почему нельзя слепо принимать «исправление» ИИ.

### 4.1. Что такое ошибка и почему программа «падает»

Когда ты слышишь «программа упала», это значит: во время выполнения возникла ситуация, с которой код не смог справиться, и Python остановил выполнение. В Python такие ситуации называются **исключениями (exceptions)**. Важно различать два больших класса проблем.

**Синтаксические ошибки (SyntaxError)** — код написан неправильно с точки зрения грамматики языка, и Python даже не может его запустить. Пример: забытое двоеточие.

```python
def srednee(chisla)      # забыли двоеточие в конце
    return sum(chisla) / len(chisla)
```

Python ответит ещё до запуска:

```
  File "main.py", line 1
    def srednee(chisla)
                       ^
SyntaxError: expected ':'
```

**Ошибки времени выполнения (исключения)** — код синтаксически верен и запускается, но во время работы натыкается на ситуацию, которую не предусмотрел программист: деление на ноль, обращение к несуществующему элементу списка, сложение числа со строкой. Именно с такими ошибками ты будешь работать чаще всего, и именно их умеет разбирать ИИ.

Запомни ключевую мысль: **«запускается» не равно «работает правильно».** Программа может не падать и при этом давать неверный результат — это **логическая ошибка**, и она опаснее всего, потому что не выдаёт себя красным текстом. Логические ошибки ловятся не traceback'ом, а тестами (раздел 4.6).

### 4.2. Как читать сообщение об ошибке (traceback)

**Traceback** — это не «страшный красный текст», а карта, которая показывает, где и что сломалось. Слово переводится как «прослеживание назад»: Python показывает цепочку вызовов, которая привела к ошибке. Читать traceback нужно **снизу вверх**: самая важная строка — последняя, в ней тип ошибки и её описание.

Разберём настоящий пример. Пусть есть программа:

```python
def srednee(chisla):
    return sum(chisla) / len(chisla)

ocenki = []
print(srednee(ocenki))
```

Запускаем — и получаем traceback:

```
Traceback (most recent call last):
  File "main.py", line 5, in <module>
    print(srednee(ocenki))
          ^^^^^^^^^^^^^^^^
  File "main.py", line 2, in srednee
    return sum(chisla) / len(chisla)
           ~~~~~~~~~~~~^~~~~~~~~~~~~~
ZeroDivisionError: division by zero
```

Прочитаем эту карту по частям:

1. **`Traceback (most recent call last)`** — заголовок: ниже идёт цепочка вызовов, и самый последний (тот, где всё сломалось) — внизу.
2. **`File "main.py", line 5, in <module>`** — ошибка началась со строки 5 основного кода: мы вызвали `srednee(ocenki)`.
3. **`File "main.py", line 2, in srednee`** — внутри функции `srednee`, на строке 2, и произошёл сбой. Стрелочки `~~~~^~~~` под кодом указывают на конкретное место — деление.
4. **`ZeroDivisionError: division by zero`** — **самая важная строка.** Слева — *тип ошибки* (`ZeroDivisionError`), справа после двоеточия — *описание* («деление на ноль»).

Уже из последней строки понятна причина: где-то делим на ноль. А цепочка выше говорит, *где* именно: строка 2, выражение `sum(chisla) / len(chisla)`. Соединяем: список пустой → `len(chisla)` равно 0 → деление на ноль. Причину мы нашли, ещё не открыв ИИ.

Вот мини-алгоритм чтения любого traceback:
- **читай снизу вверх** — последняя строка важнее всего;
- **возьми тип ошибки** (слово перед двоеточием) — он сразу сужает круг причин;
- **прочитай описание** (после двоеточия) — часто там прямой ответ;
- **найди свою строку** — Python указывает файл и номер строки;
- **посмотри на стрелки** `^^^` — они указывают на конкретное место в строке.

### 4.3. Словарь самых частых ошибок Python

Чтобы быстро понимать traceback, держи в голове «словарик» типичных исключений. Это сэкономит тебе обращения к ИИ — половину ошибок ты будешь узнавать в лицо [9].

| Тип ошибки | Что значит | Типичная причина |
|---|---|---|
| `ZeroDivisionError` | деление на ноль | делитель оказался 0 (часто `len` пустого списка) |
| `TypeError` | операция над несовместимыми типами | сложили число со строкой, вызвали не то |
| `ValueError` | тип верный, но значение недопустимо | `int("abc")` — строка не похожа на число |
| `IndexError` | обращение к несуществующему индексу списка | `spisok[5]`, когда в списке 3 элемента |
| `KeyError` | обращение к несуществующему ключу словаря | `d["age"]`, а такого ключа нет |
| `NameError` | использована необъявленная переменная | опечатка в имени переменной |
| `AttributeError` | у объекта нет такого метода/свойства | `"text".apend(...)` вместо `append` |
| `FileNotFoundError` | файл не найден | неверный путь к файлу |

Посмотрим на `TypeError` вживую:

```python
vozrast = input("Введите возраст: ")   # input всегда возвращает строку
print(vozrast + 5)
```

```
Traceback (most recent call last):
  File "main.py", line 2, in <module>
    print(vozrast + 5)
          ~~~~~~~~^~~
TypeError: can only concatenate str (not "int") to str
```

Описание прямо говорит: к строке (`str`) можно прибавить только строку, а не число (`int`). Причина: `input()` вернул строку `"30"`, а мы пытаемся сложить её с числом `5`. Решение очевидно из самого описания — превратить строку в число: `int(vozrast) + 5`. Видишь, как тип ошибки + описание почти всегда указывают путь к решению?

### 4.4. Алгоритм отладки: ошибка → traceback → гипотеза с ИИ → проверка → тест

Теперь соберём всё в единый рабочий алгоритм. Открой схему «Алгоритм отладки с ИИ» (приложение А) — она показывает этот путь визуально: ошибка → читаем traceback → гипотеза с ИИ → проверка → тест, а пунктирная жёлтая стрелка показывает возврат: если гипотеза оказалась неверной, формулируем новую. Разберём каждый шаг подробно.

**Шаг 1. Ошибка — программа упала.** Не паникуй и не трогай код. Первое, что ты делаешь, — читаешь сообщение, а не редактируешь строки. Самая частая и дорогая ошибка новичка — начать чинить раньше, чем понял причину.

**Шаг 2. Читаем traceback.** Применяешь алгоритм из раздела 4.2: смотришь тип ошибки и описание (последняя строка), находишь свою строку. Очень часто на этом шаге причина уже ясна, и ИИ не нужен вовсе. Например, `ZeroDivisionError` в строке деления → ищи делитель, который стал нулём.

**Шаг 3. Гипотеза с ИИ.** Если по traceback причина непонятна (ошибка в чужом коде, незнакомая библиотека, длинная цепочка вызовов) — подключаешь ИИ. Ты вставляешь *текст ошибки* и *фрагмент кода* и просишь объяснить причину. Важно: ИИ выдвигает **гипотезу** — предположение о причине, а не истину в последней инстанции. Как безопасно формулировать запрос — в разделе 4.5.

**Шаг 4. Проверка: это причина?** Это ключевой шаг, который новички пропускают. Ты не «веришь» ИИ — ты *проверяешь* его гипотезу. Способы проверки: прочитать код и сопоставить с объяснением; добавить временный вывод значений (`print(len(chisla))`); воспроизвести ошибку на минимальном примере; свериться с документацией, существует ли вообще функция, которую советует ИИ. Если гипотеза подтвердилась — идёшь дальше. Если нет (жёлтая стрелка на схеме) — возвращаешься к шагу 3 и формулируешь новую гипотезу.

**Шаг 5. Тест: исправил → проверил.** Ты не просто исправляешь код, а закрываешь баг **тестом**, чтобы он больше не вернулся. Тест на тех самых данных, которые ломали программу (пустой список), — гарантия, что баг устранён по-настоящему.

Применим алгоритм к примеру с `ZeroDivisionError` из раздела 4.2:

```python
def srednee(chisla):
    return sum(chisla) / len(chisla)
```

- **Шаг 1–2:** прочитали traceback → `ZeroDivisionError: division by zero`, строка деления.
- **Шаг 3:** гипотеза (можно и без ИИ): список пуст, `len([])` = 0.
- **Шаг 4:** проверяем — да, мы вызвали `srednee([])`. Гипотеза верна.
- **Шаг 5:** исправляем причину и пишем тест.

Исправление, которое устраняет *причину*:

```python
def srednee(chisla):
    if len(chisla) == 0:
        raise ValueError("Список пустой: среднее не определено")
    return sum(chisla) / len(chisla)
```

Здесь мы не «проглотили» ошибку, а явно сказали: на пустом списке среднее не определено, и сообщили об этом понятным сообщением. Это честное поведение — программа предупреждает о неверных данных, а не выдаёт мусор. Тест на этот случай напишем в разделе 4.7.

### 4.5. Как правильно и безопасно спрашивать ИИ об ошибке

ИИ хорош в двух вещах: **объяснить** незнакомую ошибку простыми словами и **подсказать** вероятную причину или идею. Но качество ответа зависит от того, что ты дал на вход, и от того, что ты *не* дал.

**Что включить в запрос:**
- полный текст traceback (а не «у меня ошибка»);
- минимальный фрагмент кода, в котором она возникает;
- что ты ожидал получить и что получил;
- что ты уже пробовал.

**Что НЕЛЬЗЯ вставлять в запрос:**
- реальные пароли, токены, API-ключи, строки подключения к базе;
- персональные данные (ФИО, ИИН, телефоны, адреса реальных людей);
- внутренние секреты компании.

Это прямое требование безопасности и закона: публичный ИИ-сервис — это посторонняя сторона, и передавать ему персональные данные без оснований нельзя. В Казахстане это регулирует **Закон РК «О персональных данных и их защите» № 94-V** [4]. Если в коде, который ты хочешь показать ИИ, есть секрет или реальные данные — **обезличь** их: замени пароль на `"PASSWORD"`, ИИН на `000000000000`, реальные имена на «Клиент 1», возьми тестовые (выдуманные) числа.

Сравни два запроса к ИИ.

Плохой запрос (мало контекста, есть секрет):
> «Почему не работает? `db.connect("postgres://admin:Qwerty123@10.0.0.5/clients")` падает.»

Хороший запрос (есть контекст, секрет убран):
> «При запуске функции получаю ошибку. Вот traceback: `ZeroDivisionError: division by zero` на строке `return sum(chisla) / len(chisla)`. Вот функция: [код без реальных данных]. Я передаю пустой список. Объясни причину и предложи, как корректно обработать пустой ввод.»

Второй запрос даёт ИИ всё нужное и при этом ничем не рискует. Это и есть человекоцентричный подход к ИИ из занятия 25: ты ставишь задачу, ИИ помогает, ты проверяешь и отвечаешь за результат [7].

### 4.6. Тестирование: зачем оно нужно и какие бывают тесты

Перейдём ко второй половине темы. **Тест** — это код, который проверяет, что твой код работает правильно. Вместо того чтобы каждый раз запускать программу руками и глазами сверять результат, ты один раз пишешь тесты — и они проверяют код автоматически, причём так же тщательно при каждом запуске.

Зачем это нужно тебе, а не только «большим командам»:
- **тест ловит логические ошибки** — те, что не дают traceback, но портят результат;
- **тест защищает от возврата бага** — если ты починил баг и закрыл его тестом, повторно он не пройдёт незамеченным;
- **тест — это документация** — по тесту видно, как функция должна себя вести;
- **тест нужен в команде** — твой коллега меняет код, прогоняет тесты и сразу видит, не сломал ли он твою функцию. «Я просто запустил, вроде работает» так не умеет.

Хороший набор тестов проверяет **три вида случаев**. Открой схему «Три вида тестов для одной функции» (приложение Б) — она показывает их наглядно.

1. **Обычный ввод** — типичные, «нормальные» данные, для которых функция и писалась. Пример для среднего: `[2, 4, 6]` → ожидаем `4.0`. Смысл: «должно работать».
2. **Крайний случай (edge case)** — граничные данные, на которых код чаще всего ломается: пустой список `[]`, ноль `0`, отрицательное `-1`, один элемент `[5]`, очень большое число. Смысл: «здесь часто падает». Именно крайние случаи отделяют надёжный код от хрупкого.
3. **Некорректный ввод** — данные не того типа или формата: строка вместо числа `"текст"`, `None`, буквы там, где ожидались цифры. Смысл: «должно дать понятную ошибку, а не упасть мусором».

Возьмём функцию «среднее арифметическое списка» и распишем для неё все три вида:

| Вид теста | Данные | Что ожидаем |
|---|---|---|
| Обычный | `[2, 4, 6]` | `4.0` |
| Крайний | `[5]` (один элемент) | `5.0` |
| Крайний | `[]` (пусто) | понятная ошибка (`ValueError`) |
| Некорректный | `["a", "b"]` | ошибка типа (`TypeError`) |

Эта таблица — готовый «скелет» тестов для любой функции: сначала думаешь про обычные данные, потом про границы, потом про «мусор» на входе.

### 4.7. Как написать тест: от assert до pytest

Самый простой тест в Python — оператор **`assert`**. Он проверяет условие: если истинно — молчит, если ложно — роняет программу с `AssertionError`. Это уже полноценная автоматическая проверка.

```python
def srednee(chisla):
    if len(chisla) == 0:
        raise ValueError("Список пустой")
    return sum(chisla) / len(chisla)

# Обычный случай
assert srednee([2, 4, 6]) == 4.0
# Крайний: один элемент
assert srednee([5]) == 5.0
print("Все обычные тесты прошли")
```

Если все `assert` верны, увидишь «Все обычные тесты прошли». Если хотя бы один нарушится, получишь:

```
Traceback (most recent call last):
  File "main.py", line 7, in <module>
    assert srednee([2, 4, 6]) == 4.0
AssertionError
```

Это и есть тест: он *сам* сказал, что результат не тот, какой ожидался.

Как проверить, что функция **правильно роняет ошибку** на крайнем/некорректном вводе? Для этого ловим исключение и убеждаемся, что оно действительно случилось:

```python
# Крайний случай: пустой список должен дать ValueError
try:
    srednee([])
    print("ОШИБКА ТЕСТА: исключение не возникло")
except ValueError:
    print("OK: пустой список корректно даёт ValueError")
```

На практике разработчики не пишут такие проверки руками, а используют инструмент **`pytest`** — он сам находит тесты, запускает их и красиво показывает результат [11]. Тест на pytest — это обычная функция с именем, начинающимся на `test_`:

```python
import pytest
from main import srednee

def test_obychnyy():
    assert srednee([2, 4, 6]) == 4.0

def test_odin_element():
    assert srednee([5]) == 5.0

def test_pustoy_spisok():
    with pytest.raises(ValueError):     # ждём именно ValueError
        srednee([])
```

Запуск командой `pytest` в терминале даст отчёт вида:

```
collected 3 items
test_main.py ...                                              [100%]
3 passed in 0.02s
```

Три точки — три пройденных теста. Если бы один упал, pytest показал бы, *какой* тест и *почему*. Это намного удобнее ручной проверки, и именно так тестируют код в реальных проектах. Глубоко pytest осваивать сейчас не нужно — важно понять идею: тест это код, и его можно запускать автоматически.

### 4.8. Генерация тестов с ИИ — и где здесь подвох

ИИ отлично помогает на этапе придумывания тестов. Его сильная сторона — фантазия на «ломающие» данные: он быстро предложит крайние и некорректные случаи, о которых ты мог не подумать. Типичный полезный запрос:

> «Вот функция `srednee(chisla)` — среднее арифметическое списка. Предложи набор тестов: обычные, крайние и некорректные случаи. Для каждого укажи входные данные и ожидаемое поведение.»

ИИ может вернуть, например: `[2,4,6]`, `[5]`, `[]`, `[-1,-2,-3]`, `["a","b"]`, очень большой список, список с дробями. Часть из них ты бы не вспомнил — и это ценно: именно так находят скрытые баги (вспомни мини-кейс урока: функция работала на `[1,2,3]`, но падала на пустом списке `[]`, и тест с `[]` это вскрыл).

Но есть **два подвоха**, из-за которых тесты от ИИ нельзя брать вслепую:

1. **ИИ может ошибиться в ожидаемом результате.** Он напишет `assert srednee([1,2,3,4]) == 2.4`, хотя правильный ответ `2.5`. Если ты вставишь такой тест не проверяя, ты «зафиксируешь» неверное поведение как эталон — и теперь твой *тест* врёт. Поэтому ожидаемое значение всегда проверяй сам, хотя бы устно посчитав.
2. **ИИ может предложить тест под несуществующее поведение.** Он напишет тест, опирающийся на функцию или параметр, которых в коде нет. Такой тест либо не запустится, либо проверяет не то.

Вывод: используй ИИ как генератор *идей* для тестов (особенно крайних), но *ожидаемые результаты* и *смысл* каждого теста проверяй сам. Идеи — от ИИ, истина — от тебя.

### 4.9. Почему нельзя слепо принимать «исправление» ИИ

Это главная мысль всей темы, поэтому разберём её на конкретном коде. ИИ умеет не только объяснять ошибки, но и предлагать исправления. Беда в том, что исправление может **маскировать** баг, а не устранять его причину. Маскировка — это когда ошибка перестаёт быть видна, но проблема остаётся.

Самый частый пример маскировки — обернуть всё в `try/except` и «проглотить» любую ошибку. Вернёмся к среднему. Был баг — падение на пустом списке. ИИ (или невнимательный программист) предлагает:

```python
def srednee(chisla):
    try:
        return sum(chisla) / len(chisla)
    except:                 # ловим вообще всё
        return 0            # и тихо возвращаем ноль
```

Программа больше не падает — красный текст исчез, «починилось». Но что произошло на самом деле? Разберём, почему это плохо:

- **Ложный результат.** На пустом списке функция теперь возвращает `0`. Но среднее пустого списка — это *не ноль*, это «не определено». Программа врёт, и эта ложь молча уйдёт дальше по коду.
- **`except` без типа ловит ВСЁ.** Сюда попадут и совсем другие ошибки — например, `TypeError` от нечислового ввода `["a","b"]`. Реальный баг (передали не тот тип) будет проглочен и спрятан под тем же `0`. Ты потеряешь даже шанс его заметить.
- **Причина не устранена.** Источник проблемы — что функцию вызывают с пустым/неверным вводом — остался. Баг просто стал невидимым и всплывёт позже, когда тот самый `0` приведёт к неверному отчёту.

Сравни это с честным исправлением из раздела 4.4: там мы проверили причину (`if len(chisla) == 0`) и явно сообщили об ошибке через `raise ValueError`. Разница принципиальная:

| Маскировка (плохо) | Исправление причины (хорошо) |
|---|---|
| `try/except: return 0` — прячет ошибку | `if len==0: raise ValueError` — называет проблему |
| ловит все ошибки подряд | реагирует на конкретный случай |
| даёт ложный результат | даёт понятное сообщение или корректный результат |
| баг вернётся позже | баг устранён по-настоящему |

Это не значит, что `try/except` — зло. Это нужный инструмент, но применять его надо **точечно**: ловить *конкретный* тип ошибки и делать с ним *осмысленное* действие, а не «глотать всё подряд». Например:

```python
nomer = input("Введите число: ")
try:
    chislo = int(nomer)
except ValueError:               # ловим ТОЛЬКО ошибку преобразования
    print("Это не число, попробуй ещё раз")
    chislo = None
```

Здесь мы заранее знаем, что пользователь может ввести не число, ловим именно `ValueError` и реагируем понятно — это обработка, а не маскировка.

Главный принцип темы: **ИИ предлагает — проверяешь и решаешь ты.** Прежде чем принять исправление ИИ, задай три вопроса: *устраняет ли оно причину или только прячет симптом? не глотает ли оно другие ошибки заодно? даёт ли честный результат или подменяет его «заглушкой»?* Если хотя бы на один ответ «плохой» — исправление принимать нельзя. И финальная проверка любого исправления — **прогнать тесты** трёх видов: если на крайнем и некорректном вводе всё ведёт себя осмысленно, баг закрыт по-настоящему.

## 5. Разобранные примеры и мини-кейсы

**Пример 1 — деление на ноль (полный цикл отладки).**
Дан код:

```python
def delenie(a, b):
    return a / b

print(delenie(10, 0))
```

Traceback:

```
ZeroDivisionError: division by zero
```

*Разбор по алгоритму:* тип ошибки прямо называет причину — деление на ноль (шаг 2, ИИ даже не нужен). Гипотеза: `b == 0` (шаг 3). Проверка: да, передали `0` (шаг 4). Честное исправление и тест (шаг 5):

```python
def delenie(a, b):
    if b == 0:
        raise ValueError("Делить на ноль нельзя")
    return a / b

# тесты
assert delenie(10, 2) == 5.0          # обычный
try:
    delenie(10, 0)                    # крайний: ждём ошибку
    assert False, "должно было упасть"
except ValueError:
    pass
```

**Пример 2 — баг, который не виден без теста (мини-кейс урока, развёрнут).**
Функция суммы списка:

```python
def summa(spisok):
    itog = 0
    for x in spisok:
        itog = x          # БАГ: присваивание вместо += 
    return itog
```

На `[1, 2, 3]` функция вернёт `3` — и это *случайно* совпало бы с длиной, но не с суммой; на самом деле она вернёт последний элемент. Программа **не падает** — traceback'а нет. Это логическая ошибка, и поймать её можно только тестом:

```python
assert summa([1, 2, 3]) == 6     # AssertionError: вернулось 3, а не 6
```

*Разбор:* «запускается» не значит «работает». Тест с известным правильным ответом (`6`) сразу вскрыл ложь. Исправление: `itog = itog + x` (или `itog += x`). Урок мини-кейса: без теста такой баг легко уезжает в продакшн.

**Пример 3 — ИИ предложил маскирующее «исправление».**
Студент показал ИИ падающую функцию `srednee`, и ИИ предложил вариант с `try/except: return 0` из раздела 4.9. Студент почти вставил его — программа же перестаёт падать.
*Разбор:* это маскировка. Проверка тремя вопросами: причину не устраняет (да), глотает все ошибки (да), даёт ложный `0` (да) — три «плохо», принимать нельзя. Правильный путь — `raise ValueError` на пустом вводе и тесты на `[]` и `["a","b"]`. Здесь сработал ключевой навык темы: студент не принял ответ ИИ вслепую, а проверил его.

## 6. Частые ошибки и как их избежать

- **Ошибка:** чинить код «методом тыка», не читая traceback. **Почему:** меняя строки наугад, теряешь время и не понимаешь причину — баг возвращается. **Как правильно:** сначала прочитай последнюю строку traceback (тип + описание), найди свою строку, выдвини гипотезу.
- **Ошибка:** принимать исправление ИИ вслепую. **Почему:** ИИ может замаскировать баг (`try/except: return 0`) или ошибиться. **Как правильно:** проверь тремя вопросами (устраняет причину? не глотает другие ошибки? даёт честный результат?) и прогони тесты.
- **Ошибка:** вставлять в запрос к ИИ секреты и реальные данные. **Почему:** это утечка и нарушение Закона РК № 94-V [4]. **Как правильно:** обезличь — замени пароли, ключи, ИИН и имена на условные/тестовые значения.
- **Ошибка:** тестировать только «обычный» ввод. **Почему:** код чаще всего ломается на границах (пусто, ноль, отрицательное) и на «мусоре». **Как правильно:** для каждой функции пиши все три вида тестов: обычный, крайний, некорректный.
- **Ошибка:** считать «программа запустилась» доказательством правильности. **Почему:** логические ошибки не дают traceback, но портят результат. **Как правильно:** проверяй результат тестом с заранее известным правильным ответом.
- **Ошибка:** доверять ожидаемому результату, который придумал ИИ. **Почему:** ИИ может ошибиться в эталонном значении и «зафиксировать» неверное поведение. **Как правильно:** считай ожидаемый результат сам, ИИ используй только для идей по входным данным.

## 7. Памятка / практический алгоритм

Короткий чек-лист отладки и тестирования с ИИ — можно держать перед глазами:

1. **Не трогай код — прочитай traceback.** Последняя строка: тип ошибки + описание. Найди свою строку.
2. **Узнай ошибку в лицо** по словарику (раздел 4.3) — часто причина ясна без ИИ.
3. **Если непонятно — спроси ИИ.** Вставь traceback и фрагмент кода. Убери секреты и реальные данные.
4. **Проверь гипотезу ИИ.** Сопоставь с кодом, выведи значения, сверься с документацией. Не та причина — новая гипотеза.
5. **Исправляй причину, а не симптом.** Никаких «проглатывающих» `try/except` без типа.
6. **Закрой баг тестом.** Напиши тесты трёх видов: обычный, крайний, некорректный. Ожидаемый результат посчитай сам.
7. **Прогони тесты.** Зелено — баг устранён по-настоящему. Красно — читай traceback и начинай сначала.

## 8. Краткие итоги

- Сначала читай traceback, а не меняй код наугад: последняя строка (тип + описание) часто содержит причину.
- Читай traceback снизу вверх; знай в лицо частые ошибки (`ZeroDivisionError`, `TypeError`, `IndexError` и др.).
- Алгоритм отладки: ошибка → traceback → гипотеза с ИИ → проверка → тест; неверная гипотеза → новая гипотеза.
- ИИ объясняет ошибки и придумывает тесты — но в запрос нельзя вставлять секреты и реальные данные.
- Тестируй три вида случаев: обычный, крайний и некорректный ввод; именно крайние случаи отделяют надёжный код от хрупкого.
- «Запускается» не значит «работает правильно» — логические ошибки ловятся только тестами.
- Не принимай исправление ИИ вслепую: оно может маскировать баг (`try/except: return 0`), а не устранять причину.
- За код, исправления и тесты отвечаешь ты, а не ИИ; финальная проверка любого исправления — прогон тестов.

## 10. Глоссарий

- **assert** — оператор Python, проверяющий условие; при ложном условии роняет программу с `AssertionError`.
- **AssertionError** — ошибка, возникающая, когда условие `assert` ложно; так «сообщает» о себе непройденный тест.
- **Edge case (крайний случай)** — граничные данные (пусто, ноль, отрицательное, один элемент), на которых код чаще всего ломается.
- **Исключение (exception)** — ошибка времени выполнения, прерывающая программу, если она не обработана.
- **Логическая ошибка** — программа не падает, но даёт неверный результат; ловится тестами, а не traceback'ом.
- **Маскировка бага** — сокрытие ошибки (например, через «проглатывающий» `try/except`) без устранения её причины.
- **Некорректный ввод** — данные не того типа/формата (строка вместо числа, `None`); функция должна дать понятную ошибку.
- **Обезличивание** — удаление из данных секретов и опознающих признаков перед отправкой ИИ.
- **Отладка (debugging)** — поиск и устранение причины ошибки в коде.
- **pytest** — инструмент Python для автоматического запуска и проверки тестов.
- **Синтаксическая ошибка (SyntaxError)** — нарушение грамматики языка; программа не запускается вовсе.
- **Тест** — код, автоматически проверяющий, что другой код работает правильно.
- **Traceback (трассировка)** — сообщение об ошибке: тип, цепочка вызовов, файл, строка, описание.
- **try/except** — конструкция обработки исключений; применять точечно, ловя конкретный тип ошибки.

## 12. Приложения

- **Приложение А.** Схема «Алгоритм отладки с ИИ» — `assets/28_shema-algoritm-otladki.svg`. Читай её слева направо: ошибка → читаем traceback → гипотеза с ИИ → проверка («это причина?») → тест. Подпись внизу: ИИ помогает придумать «ломающие» данные для крайних и некорректных случаев.
