## 1. Введение: зачем тебе эта тема

Начну с картинки, которую видел живьём десятки раз. Человек ведёт таблицу учёта — курс доллара, остатки, что угодно. Каждое утро он открывает сайт, находит нужное число, переписывает в свою табличку. Десять секунд, мелочь. А теперь посчитай: десять секунд каждый рабочий день — это час в год чистого времени плюс однажды забытая строка, однажды перепутанная дата, однажды скопированное «не то». И всё это — ради числа, которое сервис и так отдаёт бесплатно и мгновенно, если попросить его правильно.

А рядом стоит приложение погоды у него же в телефоне. Оно ничего не переписывает руками — само, каждые несколько минут, ходит на сервер за свежими данными и рисует их на экране. Бот в мессенджере сам присылает напоминания. Сайт магазина сам спрашивает у банка, прошла ли оплата, и сам открывает доступ к товару. Никто не сидит и не «переписывает числа». За людей это делают программы, которые умеют разговаривать друг с другом.

Вот этот разговор программ и происходит через **API**. Если совсем коротко — API это способ одной программы вежливо попросить что-то у другой и получить ответ в понятном, разбираемом виде. Пока ты не понимаешь, как это устроено, ты остаёшься тем самым человеком с таблицей: ты *пользователь* готовых программ, и не более. Как только ты понял API, ты перешёл на другую сторону — ты тот, кто *соединяет* программы между собой и сваливает на машину нудную повторяющуюся работу [1].

Почему это важно именно тебе как будущему разработчику? Потому что современные программы почти никогда не работают в одиночку. Открой любое серьёзное приложение и мысленно разбери его на части: карту оно берёт у картографического сервиса, оплату проводит через платёжную систему, уведомления шлёт через чужой сервис рассылок, а ответ на вопрос пользователя иногда просит у ИИ-сервиса. Всё это — API. И наоборот: твоё собственное приложение однажды будет отдавать данные другим через свой API. Получается, «читать и составлять запросы к API» — это не узкая тема одного занятия, а базовый кирпич профессии. Без него современную разработку буквально не собрать [2; 3].

И теперь хорошая новость, которая снимает страх. Чтобы сделать первый запрос, тебе не нужно знать ни одного языка программирования. Простейший запрос к API делается прямо в адресной строке браузера — так же, как ты открываешь обычный сайт. Больше того: в этой теме есть настоящий рабочий адрес, который ты можешь вставить в браузер прямо сейчас и увидеть живой ответ сервиса погоды. Никаких ключей, регистраций и установок. В этой теме ты разберёшься, из чего состоит запрос, как читать ответ, чем отличаются GET и POST и как не «спалить» свой ключ доступа — на живых примерах, а не на выдуманных [9].

## 3. Ключевые понятия

- **API** — набор правил, по которым одна программа обращается к другой: что можно попросить и в каком виде придёт ответ [1].
- **REST API** — самый распространённый вид API в интернете: обращение идёт по обычному веб-адресу (URL) и работает поверх протокола HTTP [1].
- **URL (адрес)** — строка, указывающая, *куда* отправляется запрос.
- **HTTP-метод** — слово, говорящее, *что* мы хотим сделать: `GET` (получить), `POST` (отправить/создать) и другие.
- **Параметры запроса** — уточнения к запросу (например, координаты или дата), которые пишут после `?`.
- **JSON** — текстовый формат данных в виде пар «ключ: значение», в котором чаще всего приходит ответ [3].
- **Ключ (поле) JSON** — имя, по которому из ответа достают нужное значение.
- **API-ключ (токен)** — секретный «пароль» твоей программы для доступа к сервису.
- **Код ответа (HTTP-статус)** — число, показывающее, успешно ли выполнен запрос (например, `200` — успех, `404` — не найдено).
- **Эндпоинт (точка доступа)** — конкретный адрес внутри API, отвечающий за одну операцию.
- **requests** — популярная библиотека Python, которой удобно отправлять HTTP-запросы из кода.

Полные определения собраны в глоссарии в конце материала.

## 4. Подробная теория

### 4.1. Что такое API: аналогия с кафе и меню

Начнём с определения, а потом разберём его по частям — так понятнее.

> **API** (Application Programming Interface, «интерфейс программирования приложений») — это набор правил, по которым одна программа обращается к другой: что именно можно попросить и в каком виде придёт ответ [1].

Звучит абстрактно, поэтому держи в голове аналогию — она объясняет суть лучше любого определения. **Представь, что ты пришёл в кафе.** Ты не идёшь на кухню, не зажигаешь плиту, не ищешь, где лежат продукты. Ты садишься за стол, берёшь *меню*, выбираешь блюдо и говоришь *официанту*: «мне, пожалуйста, вот это». Через время официант приносит готовое блюдо. Тебе не важно, как именно повар его готовил, какие у него ножи и сколько лет плите. Тебе важно одно: **есть меню (список того, что можно заказать) и есть понятный способ сделать заказ и получить результат.**

Вот это меню и есть API. Сервис (кафе) говорит тебе: «вот список того, что я умею отдавать, вот как об этом просить, вот в каком виде придёт ответ». Ты (твоя программа) не знаешь и не должен знать, как сервис устроен внутри — какие у него базы, на каком языке он написан, где стоят его сервера. Ты знаешь только две вещи: **что попросить и что получишь.** Посмотри на схему «API как меню» (приложение А) — там запрос и ответ изображены как заказ официанту и принесённое блюдо.

Почему эта аналогия так важна? Потому что она объясняет главное свойство API — **скрытие сложности**. Картографический сервис обрабатывает спутниковые снимки, строит маршруты, считает пробки — это гигантская система. Но тебе, чтобы показать карту в своём приложении, всё это уметь не нужно. Ты просто «делаешь заказ по меню»: просишь карту нужного города — и получаешь. API превращает чужую огромную работу в один простой запрос. Именно поэтому один разработчик в одиночку собирает приложение, которое показывает погоду, карты, курсы валют и принимает оплату: всю тяжёлую работу делают чужие сервисы, а он лишь грамотно «заказывает» у них по их меню [2].

И ещё один вывод из аналогии, важный для практики: меню — это **договор**. Кафе обещает: закажешь блюдо из меню так, как написано, — получишь именно его. Так же и API: составил запрос по правилам сервиса — получишь предсказуемый ответ. Нарушил правила (попросил то, чего в меню нет, или напутал в формулировке) — получишь не блюдо, а сообщение об ошибке. Поэтому работа с любым API всегда начинается с чтения «меню» — *документации* сервиса, где чёрным по белому написано, что и как можно у него просить. Пропустишь этот шаг — будешь гадать вслепую и злиться, что «не работает».

### 4.2. REST API: API, который работает через веб-адрес

API бывают разные, но в вебе тебе почти всегда встретится один тип — **REST API**. Его особенность в том, что обращение к нему выглядит как обычный веб-адрес, а работает оно поверх того же протокола **HTTP**, по которому твой браузер открывает сайты [1].

Это удобно для понимания: ты *уже* пользуешься HTTP каждый день, просто не задумываешься. Когда ты вводишь адрес сайта и жмёшь Enter, браузер отправляет на сервер запрос «дай мне эту страницу» и получает ответ — HTML-страницу. REST API работает точно так же, только вместо красивой страницы для человека он отдаёт «сырые» данные для программы — обычно в формате JSON. Проще всего запомнить так: **REST API — это сайт для программ, а не для людей.**

Из этого следует приятная вещь, и я хочу, чтобы ты сделал это руками прямо сейчас. Самый простой запрос к REST API можно выполнить в браузере. Если адрес API возвращает данные методом GET, ты просто вставляешь его в адресную строку, жмёшь Enter — и видишь ответ JSON прямо на экране. Вот **настоящий бесплатный адрес** сервиса погоды Open-Meteo (координаты `43.24, 76.89` — это Алматы), ключ не нужен:

```
https://api.open-meteo.com/v1/forecast?latitude=43.24&longitude=76.89&current=temperature_2m
```

Вставь его в браузер и нажми Enter. Тебе придёт примерно такой ответ (числа будут свои — погода-то настоящая, живая):

```json
{
  "latitude": 43.25,
  "longitude": 76.875,
  "timezone": "GMT",
  "current": {
    "time": "2026-07-01T09:00",
    "interval": 900,
    "temperature_2m": 27.4
  },
  "current_units": {
    "temperature_2m": "°C"
  }
}
```

Вот это и есть API вживую: ты отправил запрос (открыл адрес), сервис вернул JSON, а нужное тебе число — температура — лежит внутри по пути `current → temperature_2m`. Никакого кода, никакой регистрации. Позже, когда запросы станут сложнее, помогут инструменты вроде Postman или код на Python (к нему мы придём в разделе 4.5), но начать можно ровно так — в браузере.

Запомни связку понятий, это каркас всей темы: **REST API → обращение по URL → поверх HTTP → ответ в JSON.**

### 4.3. Из чего состоит запрос: URL, эндпоинт, метод, параметры

Любой запрос к REST API разбирается на части, как предложение по членам. Возьмём наш живой адрес Open-Meteo и разложим его. Для наглядности запишем его с явным методом (в браузере метод не пишут, но в коде и в Postman — да):

```
GET https://api.open-meteo.com/v1/forecast?latitude=43.24&longitude=76.89&current=temperature_2m
```

В нём спрятаны три составные части. Посмотри на схему «API как меню» (приложение А) — там запрос разложен ровно на эти блоки.

**1. URL (адрес) — *куда* обращаемся.**
Это часть до знака вопроса: `https://api.open-meteo.com/v1/forecast`. Она говорит, к какому сервису и к какой его функции мы идём. Разберём ещё мельче:
- `https://` — протокол. Буква «s» (secure) означает защищённое, шифрованное соединение. Для API это важно — данные (а иногда и ключи) не должны идти по сети открытым текстом.
- `api.open-meteo.com` — адрес сервера (домен). Часто у API он начинается со слова `api`, чтобы отличать его от обычного сайта для людей.
- `/v1/forecast` — путь к конкретной функции, это и есть **эндпоинт** (точка доступа). Здесь `v1` — версия API, `forecast` — «прогноз/текущая погода». У одного сервиса эндпоинтов может быть много: у Open-Meteo, например, есть отдельные адреса для качества воздуха и для исторических данных. Каждый эндпоинт — это отдельная «строчка в меню».

**2. Метод — *что* делаем.**
В примере это слово `GET` в начале. Метод сообщает сервису намерение: хотим *получить* данные, *отправить* новые, *изменить* существующие или *удалить*. Самые частые — GET и POST, им посвящён раздел 4.5. В адресной строке браузера метод не виден (браузер всегда шлёт GET), но в коде или в Postman его указывают явно.

**3. Параметры — *уточнения* к запросу.**
Это всё, что после знака вопроса: `latitude=43.24&longitude=76.89&current=temperature_2m`. Параметры уточняют, что именно мы хотим. Здесь мы говорим: погоду не вообще, а по координатам Алматы (`latitude`, `longitude`), и нам нужен показатель `temperature_2m` (температура на высоте 2 метра). Правила записи такие:
- параметры начинаются со знака `?`;
- каждый параметр — это пара `имя=значение` (`latitude=43.24`);
- если параметров несколько, их разделяют знаком `&`, как в нашем примере — три параметра через `&`.

Соберём вслух. Запрос читается так: «*методом GET (получить данные) обращаюсь к сервису Open-Meteo, к эндпоинту forecast, уточняю: координаты Алматы и нужна текущая температура*». В ответ придёт JSON, из которого мы возьмём поле `temperature_2m`.

**Ловушка, на которую попадают почти все.** Разделитель параметров — это `&`, а не `?`. Знак `?` ставится ровно **один раз** — там, где начинаются параметры. Все следующие пары цепляются через `&`. Напишешь по ошибке `...?latitude=43.24?longitude=76.89` (два вопроса) — многие сервисы либо не поймут второй параметр, либо вернут ошибку. Правильно только так: `?latitude=43.24&longitude=76.89`. Ещё частая беда — лишний пробел внутри адреса; в URL пробелов быть не должно, поэтому текстовые значения кодируют (пробел превращается в `%20`), но с числовыми координатами эта проблема тебя не коснётся.

**Практический совет.** Видишь незнакомый запрос — раскладывай его на эти три части, и сразу поймёшь, что он делает. Сначала найди метод (что), потом базовый адрес и эндпоинт (куда), потом параметры после `?` (с какими уточнениями). Этот разбор — половина умения работать с API.

### 4.4. Формат ответа JSON и как достать из него нужное поле

Запрос ушёл — сервис прислал ответ. Чаще всего он приходит в формате **JSON** (JavaScript Object Notation). Не пугайся названия: JSON — это просто текст, организованный по нескольким простым правилам, удобный и человеку, и программе [3].

JSON строится из пар **«ключ: значение»**. Вот компактный ответ учебного сервиса погоды (это упрощённая иллюстрация формата, а не настоящий ответ Open-Meteo — у него, как ты видел в разделе 4.2, температура лежит внутри вложенного объекта `current`):

```json
{
  "city": "Almaty",
  "temp": 18,
  "condition": "sunny"
}
```

Разберём правила записи:
- весь объект заключён в фигурные скобки `{ }`;
- внутри — пары `"ключ": значение`, разделённые запятыми;
- **ключ** всегда в двойных кавычках: `"temp"`;
- **значение** может быть числом (`18`), строкой в кавычках (`"Almaty"`), логическим (`true`/`false`), а также вложенным объектом или списком.

Главное умение — **достать значение по ключу**. Это как поиск слова в словаре: знаешь ключ (`temp`) — берёшь по нему значение (`18`). В нашем примере:
- по ключу `"city"` получаешь `"Almaty"`;
- по ключу `"temp"` получаешь `18`;
- по ключу `"condition"` получаешь `"sunny"`.

На схеме «API как меню» (приложение А) этот шаг показан стрелкой «берёшь нужное поле по ключу: temp».

Теперь важное — реальные ответы почти всегда **вложенные**, и наш живой Open-Meteo как раз такой. Вернись к ответу из раздела 4.2: температура там лежит не на верхнем уровне, а внутри объекта `current`:

```json
{
  "current": {
    "time": "2026-07-01T09:00",
    "temperature_2m": 27.4
  }
}
```

Здесь значение по ключу `"current"` — это **вложенный объект** (ещё одни фигурные скобки). Чтобы взять температуру, идёшь по шагам: сначала берёшь `current`, потом внутри него — `temperature_2m`. Получается путь `current → temperature_2m`, значение `27.4`. Думай об этом как о вложенных папках: чтобы добраться до файла, открываешь сначала внешнюю папку, потом внутреннюю. Запомни этот путь — ровно его мы используем в коде в разделе 4.5.

А бывает, что сервис возвращает **список** (массив) — несколько однотипных объектов в квадратных скобках `[ ]`:

```json
{
  "city": "Almaty",
  "forecast": [
    { "day": "today", "temp": 18 },
    { "day": "tomorrow", "temp": 21 }
  ]
}
```

Здесь `forecast` — список из двух элементов. Программа пройдёт по нему циклом и возьмёт температуру каждого дня. Списки появляются, когда данных «много однотипных»: список товаров, список городов, список записей открытых данных, почасовой прогноз на сутки.

**Алгоритм чтения любого ответа JSON:**
1. найди фигурные скобки `{ }` — это границы объекта;
2. прочитай ключи — это «названия полей», по ним понятно, что внутри;
3. найди ключ, который тебе нужен;
4. возьми его значение; если значение — вложенный объект `{ }` или список `[ ]`, спускайся внутрь по тому же правилу.

Это умение — фундамент всей работы с API: какой бы сервис ты ни взял, после запроса тебе всегда нужно «прочитать ответ и вытащить нужное поле».

### 4.5. Рабочий код: запрос к API из Python через requests

Открыть URL в браузере — отлично для проверки, но автоматизация живёт в коде: код можно запустить по расписанию и повторять хоть каждое утро. Самый ходовой инструмент в Python — библиотека **requests**. Разберём короткий, но полностью рабочий скрипт: он делает GET-запрос к тому самому Open-Meteo (ключ не нужен), проверяет, что ответ успешный, и достаёт температуру.

```python
import requests

url = "https://api.open-meteo.com/v1/forecast?latitude=43.24&longitude=76.89&current=temperature_2m"

r = requests.get(url)               # отправляем GET-запрос

if r.status_code == 200:            # 200 — запрос успешен, данные пришли
    data = r.json()                 # разбираем JSON-ответ в словарь Python
    temp = data["current"]["temperature_2m"]   # спускаемся: current -> temperature_2m
    print("Температура в Алматы:", temp, "°C")
else:
    print("Ошибка запроса, статус:", r.status_code)
```

Разберём построчно — это тот же навык «читать вывод», только для кода:
- `import requests` — подключаем библиотеку. Если её нет, ставится одной командой в терминале: `pip install requests` (это как раз из занятия про командную строку — установка зависимостей одной строкой).
- `url = "..."` — тот самый живой адрес; можешь сначала открыть его в браузере и убедиться, что сервис отвечает.
- `r = requests.get(url)` — библиотека сама отправляет GET-запрос и складывает ответ в переменную `r`.
- `if r.status_code == 200:` — **сначала проверяем статус, потом берём данные.** `200` означает успех. Об этом весь раздел 4.7 — это не формальность, а защита от того, чтобы «доставать температуру» из сообщения об ошибке.
- `data = r.json()` — превращает JSON-текст в обычный словарь Python, с которым удобно работать.
- `data["current"]["temperature_2m"]` — достаём значение по тому самому пути `current → temperature_2m`, который мы разобрали в 4.4. Обрати внимание: путь в коде повторяет структуру JSON один в один.
- `else:` — если статус не `200`, честно сообщаем об ошибке, а не делаем вид, что всё хорошо.

Что напечатает скрипт при успехе:

```
Температура в Алматы: 27.4 °C
```

Хочешь погоду другого города — поменяй координаты `latitude` и `longitude` в адресе, остальное не трогай. Вот и вся магия: пять содержательных строк кода заменяют ежедневный ручной поход на сайт.

**Отдельно про POST в коде — чтобы ты видел разницу не только на словах.** GET-запрос данные *забирает*. POST-запрос данные *отправляет*, и в `requests` это выглядит так (адрес здесь условный — реальный берётся из документации конкретного сервиса):

```python
import requests

url = "https://api.example.com/messages"
payload = {"chat_id": 12345, "text": "Доброе утро! Вот твоё расписание."}

r = requests.post(url, json=payload)   # данные уходят в теле запроса, не в адресе
print("Статус отправки:", r.status_code)
```

Смотри, что поменялось: вместо `requests.get` — `requests.post`, и появился аргумент `json=payload` — это **тело запроса**, данные, которые мы отправляем. В GET такого тела нет, там всё лежало в адресе после `?`. Это и есть практическая разница между «получить» и «отправить», о которой дальше.

### 4.6. GET и POST: два главных метода

Мы уже знаем, что метод говорит сервису, *что* мы хотим сделать. Двух методов хватает для огромной части задач — это **GET** и **POST**. Разница между ними принципиальна, и путать их нельзя. Посмотри на схему «GET и POST» (приложение Б).

**GET — *получить* данные.**
- Назначение: «просто посмотреть», прочитать информацию.
- Ничего не меняет на сервере — сколько раз ни повтори запрос, данные там останутся прежними. Такое свойство называют *безопасным*: у GET нет побочных эффектов.
- Параметры передаются прямо в адресе, после `?` (как мы видели: `?latitude=43.24&...`).
- Поэтому GET можно выполнить прямо в браузере — что мы и делали с Open-Meteo.
- Примеры: узнать погоду, посмотреть курс валют, получить список товаров, прочитать запись из открытых данных.

**POST — *отправить или создать* данные.**
- Назначение: «что-то изменить или добавить» на сервере.
- Меняет состояние сервера: после POST там появляется новая запись, отправляется сообщение, оформляется заказ. Повторишь дважды — может появиться две записи.
- Данные передаются не в адресе, а в **теле запроса** (это тот самый `json=payload` из кода выше). Это удобно и важно: в теле можно передать много данных, и они не светятся в адресной строке и в логах сервера.
- В браузере «просто так» POST не сделаешь — нужен код, Postman или форма на сайте.
- Примеры: отправить сообщение в мессенджер, оформить заказ, зарегистрировать пользователя, добавить запись в базу.

Запомни **простое правило выбора**: если задача — «просто посмотреть данные», это **GET**. Если задача — «что-то изменить, создать или отправить», это **POST**. Спроси себя: «изменится ли что-то на сервере после моего запроса?» Нет — GET, да — POST.

Чтобы закрепить, сведём различия в таблицу:

| Признак | GET | POST |
|---|---|---|
| Зачем | получить (посмотреть) данные | отправить / создать данные |
| Меняет сервер? | нет | да |
| Где данные запроса | в адресе после `?` | в теле запроса |
| Можно из браузера? | да | обычно нет |
| В коде requests | `requests.get(url)` | `requests.post(url, json=...)` |
| Пример | узнать курс валют | оформить заказ |

**Почему путать опасно — реальная ловушка.** Расскажу, как это стреляет в жизни. Представь, что кто-то по незнанию сделал операцию «удалить заказ» через обычную ссылку, то есть методом GET. Что происходит дальше? GET считается «безобидным просмотром», поэтому его автоматически повторяют браузеры (при обновлении страницы), поисковые роботы, которые ходят по ссылкам, и разные предзагрузчики. И вот поисковик проходит по всем «ссылкам удаления» на странице — и молча сносит заказы один за другим. Никто не нажимал кнопку, а данных нет. Классика, из-за которой в своё время админы хватались за голову. Мораль: операции, которые *меняют* данные, никогда не вешают на GET — только POST (или DELETE). И наоборот: оформлять безобидный просмотр как POST — значит без причины усложнять себе жизнь и терять возможность просто открыть адрес в браузере.

Кроме GET и POST есть и другие методы — например, PUT (заменить данные) и DELETE (удалить). Логика та же: метод выражает намерение. Но для старта тебе хватит твёрдо различать GET и POST.

### 4.7. Проверка успешности ответа: код состояния

Запрос срабатывает не всегда: можно опечататься в адресе, сервис может лежать, ключ может оказаться неверным, лимит запросов — исчерпанным. Поэтому **прежде чем брать данные из ответа, убедись, что запрос вообще удался.** Для этого служит **код состояния (HTTP-статус)** — число, которое сервис присылает вместе с ответом. В коде мы его уже видели: `r.status_code`.

Коды сгруппированы по первой цифре, и достаточно запомнить логику групп:
- **2xx — успех.** Самый частый — `200 OK`: всё хорошо, данные пришли, можно читать JSON. Ещё встретится `201 Created` — «создано» (типичный ответ на удачный POST).
- **4xx — ошибка на твоей стороне.** `404 Not Found` — такого адреса/ресурса нет (чаще всего опечатка в URL или неверный эндпоинт); `401 Unauthorized` — не представился (нет ключа или он неверный); `403 Forbidden` — ключ есть, но прав не хватает; `400 Bad Request` — запрос составлен неправильно (например, пропущен обязательный параметр); `429 Too Many Requests` — слишком часто дёргаешь сервис, превысил лимит.
- **5xx — ошибка на стороне сервера.** `500` — у сервиса что-то сломалось внутри; `503` — сервис временно недоступен (перегружен или на обслуживании). Твой запрос мог быть верным, но сервис не справился.

Практический вывод, который надо довести до автоматизма: **сначала смотри код, потом данные.** Алгоритм такой: получил ответ → проверил, что статус из группы `2xx` → только тогда читаешь JSON и достаёшь поля. Код `4xx` — ищи ошибку у себя (адрес, параметры, ключ). Код `5xx` — проблема у сервиса, имеет смысл повторить позже (в коде для этого делают повтор с паузой). Именно поэтому в нашем скрипте стоит `if r.status_code == 200:` — если этот шаг выкинуть, программа при первой же ошибке полезет доставать `data["current"]` из ответа, в котором вместо погоды лежит `{"error": true, "reason": "..."}`, и упадёт с непонятной ошибкой. Проверка статуса — одна строка, которая экономит час разбирательств.

### 4.8. API-ключ и безопасное хранение секретов

Наш Open-Meteo работал без ключа — это редкая удача для учёбы. Многие полезные API требуют, чтобы ты «представился» — доказал, что имеешь право пользоваться сервисом. Для этого служит **API-ключ** (его ещё зовут токеном) — длинная строка вроде `a7f3c9e1b4d8...`, которую сервис выдаёт при регистрации.

**Самое важное, что нужно усвоить: API-ключ — это пароль твоей программы к сервису.** Кто знает ключ, тот обращается к сервису *от твоего имени*. Отсюда — главная и очень частая ошибка новичков: **написать ключ прямо в коде и выложить этот код в открытый доступ**, например в публичный репозиторий на GitHub. Выглядит это невинно:

```python
# ТАК ДЕЛАТЬ НЕЛЬЗЯ — ключ прямо в коде
api_key = "a7f3c9e1b4d8f2a6c1e9b7d3f5a8c2e4"
r = requests.get("https://api.service.com/data", headers={"Authorization": api_key})
```

Чем это опасно? Тем, что открытые репозитории круглосуточно прочёсывают **автоматические сканеры** — программы, которые ищут в чужом коде что-то похожее на ключи. Найденный ключ используют в считанные минуты: за твой счёт расходуют платные лимиты сервиса, крадут доступные через сервис данные, а тебя ещё и заблокируют за «подозрительную активность», которую ты не совершал. И отговорка «я выложил всего на час» не спасает — сканеры не спят, известны случаи, когда ключ подхватывали через минуты после публикации.

Как надо. Ключ хранят **отдельно от кода** — чаще всего в *переменной окружения* (специальной настройке, которую программа читает при запуске). В коде тогда стоит не сам ключ, а ссылка «возьми ключ из окружения»:

```python
import os
import requests

api_key = os.environ["API_KEY"]    # ключ берётся из окружения, а не из кода
r = requests.get("https://api.service.com/data", headers={"Authorization": api_key})
```

Теперь в файле с кодом секрета нет — его невозможно случайно опубликовать. Правила, которые стоит сделать привычкой:
- **Не пиши ключ прямо в коде.** Код видят коллеги, он попадает в репозиторий, его легко случайно выложить.
- **Храни ключ отдельно** — в переменных окружения или в защищённом хранилище.
- **Не публикуй файлы с ключами.** Файл с секретами (например `.env`) добавляют в список «не выкладывать» — в Git это `.gitignore`, — чтобы он не попал в репозиторий.
- **При утечке — сразу отзови ключ.** Засветился — не надейся, что «никто не заметит». Зайди в сервис, отзови (аннулируй) старый ключ и получи новый. Старый после этого перестанет работать, и украденная копия станет бесполезной. Это единственная надёжная реакция на утечку.

Реальный кейс, чтобы засело. Студент выложил учебный проект на GitHub, а внутри — рабочий ключ к платному ИИ-сервису прямо в коде. К вечеру пришло письмо: лимит исчерпан, с привязанной карты списаны деньги за тысячи запросов, которые он не делал. Ключ подобрал сканер за считанные часы. Спасло бы ровно одно: не класть ключ в код и добавить `.env` в `.gitignore` — минута работы против испорченного вечера и счёта.

Этот принцип шире, чем API: пароли, ключи и любые секреты *никогда* не хранят в коде и не выкладывают в открытый доступ. Привычка отделять секреты от кода — признак аккуратного разработчика. И это прямо связано с законом: если по украденному ключу утекут чужие персональные данные, это уже нарушение **Закона РК «О персональных данных и их защите» № 94-V** [4], а вопросы защиты информации в целом регулирует **Закон РК «Об информатизации» № 418-V** [5].

### 4.9. Где это применяют: автоматизация рутины

Теперь, когда ты понимаешь запрос, ответ, методы, код и ключ, соберём картину применения. API нужны там, где программы должны *обмениваться данными*, и особенно — где человек иначе делал бы это руками. Три типовых направления:

- **Получение данных (GET).** Погода, курсы валют, открытые государственные данные. В Казахстане для этого есть портал **data.egov.kz** — там опубликованы открытые наборы данных с доступом по API [9]. Вместо того чтобы вручную копировать таблицу с сайта, программа делает GET-запрос и получает свежие данные сама — ровно как наш скрипт с Open-Meteo.
- **Отправка данных (POST).** Сообщение в мессенджер, запись в таблицу или базу, отправка формы. Бот, который каждое утро сам присылает тебе расписание, использует POST к API мессенджера (мы видели скелет такого запроса в разделе 4.5).
- **Связка сервисов.** Самое мощное применение: соединить несколько API в цепочку. Например, программа делает GET к сервису курсов, берёт число, а потом делает POST в твою таблицу. Так из отдельных «кирпичей» собирается автоматизация, которая работает без человека.

**Главная идея автоматизации:** ты тратишь время *один раз* — настраиваешь запросы, — а дальше работа выполняется *сама*, каждый день, без ошибок и без твоего участия. Повторяющаяся ручная рутина превращается в фоновый процесс. Вернись к примеру из введения с курсом валют: один раз настроил GET-запрос и запись в таблицу — и больше никогда не переписываешь число руками. Тот, кто это умеет, экономит себе и команде десятки часов в год.

**ИИ-ассистент как помощник — но с проверкой.** Здесь очень кстати приходит ИИ. Ты описываешь задачу словами — «сделай запрос к такому-то API и достань поле temp» — и просишь готовый пример запроса или короткий скрипт. Но помни принцип ответственной работы с ИИ из занятия 01: ИИ может ошибиться — придумать несуществующий эндпоинт, перепутать ключ в JSON (написать `temp` там, где на самом деле `temperature_2m`), забыть проверку статуса. Поэтому его пример нужно *прочитать, сверить с документацией сервиса и протестировать* — открыть URL в браузере, запустить код, — а не вставлять вслепую [6]. ИИ ускоряет черновик, но за результат отвечаешь ты.

### 4.10. Пошаговый алгоритм работы с новым API

Чтобы не теряться перед незнакомым сервисом, держи универсальный порядок действий:
1. **Найди документацию** сервиса (его «меню»). Там описаны эндпоинты, методы, параметры и формат ответа.
2. **Пойми, что тебе нужно**, и подбери эндпоинт. Реши, какой метод подходит: смотришь данные — GET, меняешь — POST.
3. **Собери запрос:** базовый URL + эндпоинт + нужные параметры после `?` (первый параметр через `?`, остальные через `&`).
4. **Узнай, нужен ли ключ.** Если да — зарегистрируйся, получи ключ и сразу реши, где его безопасно хранить (не в коде!).
5. **Проверь запрос вживую** — для GET без ключа вставь адрес в браузер и посмотри JSON; иначе используй Postman или код.
6. **Проверь код состояния.** `2xx` — успех, идём дальше; `4xx`/`5xx` — разбираемся с ошибкой (у себя или у сервиса).
7. **Прочитай JSON и достань нужные поля** по ключам (для вложенных — спускайся внутрь по шагам, как `current → temperature_2m`).
8. **Используй данные** — выведи, запиши, передай дальше. Если автоматизируешь — настрой повтор по расписанию.

Этот алгоритм работает с любым REST API, от погоды до открытых данных РК.

## 5. Разобранные примеры и мини-кейсы

**Пример 1 — разбор живого GET-запроса по частям.**
Дан рабочий запрос:
```
GET https://api.open-meteo.com/v1/forecast?latitude=43.24&longitude=76.89&current=temperature_2m
```
*Разбор:*
- **метод** — `GET`: получаем данные, на сервере ничего не меняем;
- **URL** — `https://api.open-meteo.com/v1/forecast`: сервис Open-Meteo (`api.open-meteo.com`), эндпоинт `forecast`, защищённое соединение (`https`);
- **параметры** — `latitude=43.24`, `longitude=76.89` (координаты Алматы) и `current=temperature_2m` (нужна текущая температура); первый через `?`, остальные через `&`.
*Что вернёт:* JSON с текущей погодой по Алматы. Поскольку это GET без ключа, запрос можно вставить в адресную строку браузера и сразу увидеть ответ — сделай это.

**Пример 2 — чтение вложенного ответа JSON.**
Сервис вернул (сокращённо):
```json
{ "current": { "time": "2026-07-01T09:00", "temperature_2m": 27.4 } }
```
*Разбор:* температура лежит не на верхнем уровне, а внутри объекта `current`. Путь к ней — `current → temperature_2m`, значение `27.4`. В коде Python это ровно `data["current"]["temperature_2m"]`. Если бы ты искал `data["temperature_2m"]` на верхнем уровне — получил бы ошибку «нет такого ключа»: типичная ловушка невнимательности с вложенным JSON.

**Пример 3 — чтение плоского ответа JSON.**
Учебный сервис вернул:
```json
{ "city": "Almaty", "temp": 18, "condition": "sunny" }
```
*Разбор:* здесь всё на верхнем уровне. Температуру берёшь по ключу `temp` → `18`; погодное условие по ключу `condition` → `"sunny"`; город по ключу `city` → `"Almaty"`. Чтобы показать строку «В Алматы +18, солнечно», программа собирает её из этих трёх полей.

**Пример 4 — GET или POST.**
Две задачи: (а) получить список товаров магазина; (б) добавить в магазин новый заказ.
*Разбор:* (а) — мы *смотрим* данные, на сервере ничего не меняется → метод **GET**. (б) — мы *создаём* новую запись (заказ), состояние сервера меняется → метод **POST**, данные заказа уйдут в теле запроса (`requests.post(url, json=...)`). Контрольный вопрос «изменится ли что-то на сервере?» даёт ответ сразу: в (а) нет, в (б) да.

**Пример 5 — мини-кейс автоматизации курса валют.**
Задача: каждое утро записывать курс доллара в таблицу.
*Как было вручную:* заходишь на сайт банка, находишь курс, переписываешь в таблицу — каждый день, с риском забыть или ошибиться.
*Как через API:* один раз пишешь программу, которая по расписанию делает `GET`-запрос к сервису курсов, проверяет код ответа (`200`), берёт из JSON поле с курсом по ключу и делает запись в таблицу. Дальше всё работает само. *Разбор:* ручная рутина (10 секунд × каждый день) превратилась в разовую настройку и надёжный автоматический процесс. Это и есть смысл автоматизации через API.

**Пример 6 — читаем ошибку вместо паники.**
Скрипт вывел: `Ошибка запроса, статус: 404`.
*Разбор:* это не «код сломался», а подсказка. `404` — «не найдено», почти всегда опечатка в URL или неверный эндпоинт. Проверь адрес по документации, открой его в браузере. Был бы `401` — дело в ключе (нет или неверный); `429` — слишком часто дёргаешь сервис, сделай паузу; `500`/`503` — проблема у сервиса, повтори позже. Мораль всей темы в одном примере: **сначала прочитай статус — он говорит, где искать причину.**

## 6. Частые ошибки и как их избежать

- **Ошибка:** хранить API-ключ прямо в коде и выкладывать его на GitHub. **Почему:** ключ — пароль программы; его находят автоматические сканеры и используют от твоего имени (чужой расход, утечка, блокировка). **Как правильно:** хранить ключ отдельно (переменные окружения), файл с секретами добавить в `.gitignore`, при утечке сразу отзывать и менять [4].
- **Ошибка:** брать поля из ответа, не проверив код состояния. **Почему:** в ответе может лежать сообщение об ошибке (`404`, `401`), а не данные, и программа упадёт. **Как правильно:** сначала проверь, что статус из группы `2xx` (`if r.status_code == 200`), потом читай JSON.
- **Ошибка:** путать GET и POST. **Почему:** GET для изменения данных опасен — его случайно повторяют браузеры и поисковики, что может молча удалить или задвоить записи; а POST для простого просмотра — лишнее усложнение (не откроешь в браузере). **Как правильно:** «посмотреть» — GET, «изменить/создать» — POST; спроси себя, меняется ли что-то на сервере.
- **Ошибка:** искать значение по неверному ключу или забыть про вложенность. **Почему:** ключи чувствительны к точному написанию, а нужное поле часто лежит внутри другого объекта. **Как правильно:** сверь ключ с документацией; для вложенного JSON спускайся по шагам (`current → temperature_2m`, а не `temperature_2m` сразу).
- **Ошибка:** неправильно склеивать параметры — ставить `?` вместо `&` между ними. **Почему:** знак `?` открывает блок параметров ровно один раз, дальше пары цепляются через `&`; иначе сервис не поймёт запрос. **Как правильно:** `?latitude=43.24&longitude=76.89`, а не `?latitude=43.24?longitude=76.89`.
- **Ошибка:** доверять примеру запроса от ИИ без проверки. **Почему:** ИИ может выдумать несуществующий эндпоинт или перепутать ключ JSON. **Как правильно:** сверь с документацией сервиса, открой URL в браузере и протестируй запрос [6].

## 7. Памятка: «Прежде чем отправить запрос»

Короткий чек-лист для работы с любым API:
1. нашёл документацию (меню) сервиса;
2. выбрал метод по смыслу: смотрю — GET, меняю — POST;
3. собрал запрос: URL + эндпоинт + параметры (первый через `?`, дальше через `&`);
4. если нужен ключ — храню его отдельно от кода, не публикую, файл с секретом — в `.gitignore`;
5. проверил запрос вживую: GET — в браузере, иначе Postman/код;
6. выполнил запрос и **проверил код состояния** (нужен `2xx`);
7. прочитал JSON и достал нужные поля по ключу (для вложенных — по шагам);
8. использовал данные; если автоматизирую — настроил повтор по расписанию.

## 8. Краткие итоги

- Воспринимай API как «меню» запросов к чужой программе: ты знаешь, что попросить и что получишь, не зная устройства внутри.
- В интернете чаще всего встречается REST API — «сайт для программ»: обращение по URL поверх HTTP, ответ в JSON.
- Запрос состоит из трёх частей: URL (куда, включая эндпоинт) + метод (что) + параметры (уточнения после `?`, между ними `&`).
- Ответ обычно в JSON — пары «ключ: значение»; нужное берёшь по ключу, для вложенного спускаешься внутрь (`current → temperature_2m`).
- Простой GET можно открыть прямо в браузере; в коде на Python помогает библиотека `requests` (`requests.get`/`requests.post`).
- GET — получить (ничего не меняет), POST — отправить/создать (меняет сервер, данные в теле запроса).
- Перед чтением данных проверь код состояния: нужен успех из группы `2xx`.
- API-ключ — это пароль программы; храни его отдельно от кода и никогда не публикуй.
- Главный смысл API для тебя — автоматизация: один раз настроил запросы, дальше машина делает рутину сама.

## 10. Глоссарий

- **API** — набор правил, по которым одна программа обращается к другой: что можно попросить и в каком виде придёт ответ.
- **API-ключ (токен)** — секретная строка-«пароль» программы для доступа к сервису; хранится отдельно от кода.
- **GET** — HTTP-метод для получения данных; ничего не меняет на сервере, параметры идут в адресе.
- **HTTP** — протокол, по которому браузеры и программы обмениваются запросами и ответами с серверами.
- **JSON** — текстовый формат данных в виде пар «ключ: значение»; основной формат ответа API.
- **Код состояния (HTTP-статус)** — число об успешности запроса: `2xx` — успех, `4xx` — ошибка клиента, `5xx` — ошибка сервера.
- **Ключ (поле) JSON** — имя, по которому из ответа достают значение.
- **Метод (HTTP-метод)** — слово, выражающее намерение запроса (GET, POST, PUT, DELETE).
- **Параметры запроса** — уточнения после знака `?` в адресе, в виде пар `имя=значение`, разделённые `&`.
- **POST** — HTTP-метод для отправки/создания данных; меняет состояние сервера, данные идут в теле запроса.
- **requests** — библиотека Python для отправки HTTP-запросов из кода (`requests.get`, `requests.post`).
- **REST API** — распространённый вид API: обращение по URL поверх HTTP, ответ обычно в JSON.
- **Тело запроса** — часть POST-запроса, в которой передаются отправляемые данные (не видна в адресе).
- **URL (адрес)** — строка, указывающая, куда отправляется запрос.
- **Эндпоинт (точка доступа)** — конкретный адрес внутри API, отвечающий за одну операцию.
