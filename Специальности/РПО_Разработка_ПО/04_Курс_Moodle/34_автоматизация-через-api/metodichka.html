Это расширенный разбор темы для самостоятельного изучения. Он подробнее, чем конспект урока: здесь каждое понятие объясняется глубже, с примерами запросов, разбором JSON и пошаговыми алгоритмами. Читай его как «учебник по теме» — после него ты не просто запомнишь, что такое API, а научишься читать чужие запросы, собирать свои и видеть, где машине можно поручить рутину.

1. Введение: зачем тебе эта тема

Представь обычное утро. Ты открываешь сайт банка, смотришь курс доллара, переписываешь число в свою таблицу учёта. Десять секунд — мелочь. Но это каждый день, и однажды ты забудешь, ошибёшься цифрой или скопируешь курс не на ту дату. А теперь посмотри на своё приложение погоды: оно само, без тебя, каждые несколько минут берёт свежие данные с сервера и показывает их на экране. Бот в мессенджере сам рассылает напоминания. Сайт магазина сам спрашивает у банка, прошла ли оплата, и сам открывает доступ к товару. Никто не сидит и не «переписывает числа руками» — за людей это делают программы, которые разговаривают друг с другом.

Этот разговор программ происходит через API. Если коротко, API — это способ одной программы вежливо попросить что-то у другой и получить ответ в понятном виде. Пока ты не знаешь, как это устроено, ты остаёшься на месте того человека с таблицей: ты пользователь готовых программ. Как только ты понимаешь API, ты переходишь на другую сторону — ты тот, кто соединяет программы между собой и поручает машине нудную работу [1].

Почему это важно именно тебе как будущему разработчику? Потому что современные программы почти никогда не работают в одиночку. Любое серьёзное приложение постоянно обращается к чужим сервисам: берёт карты у картографического сервиса, принимает оплату через платёжную систему, отправляет уведомления, спрашивает у ИИ-сервиса ответ на вопрос пользователя. И наоборот — твоё собственное приложение однажды будет отдавать данные другим через свой API. Получается, что навык «читать и составлять запросы к API» — это не узкая тема одного занятия, а базовый кирпич всей профессии. Без него современную разработку буквально не представить [2; 3].

Хорошая новость: чтобы начать, тебе не нужно знать ни одного сложного языка. Самый простой запрос к API можно сделать прямо в адресной строке браузера. В этой теме ты разберёшься, из чего состоит запрос, как читать ответ, чем отличаются основные способы обращения и как не «спалить» свой ключ доступа. А заодно увидишь живые примеры на казахстанском портале открытых данных [9].

2. Что ты узнаешь и чему научишься

После изучения темы ты сможешь:

  • объяснять своими словами, что такое API и зачем он нужен, на простой аналогии;
  • разбирать структуру запроса на части — адрес (URL), метод и параметры;
  • читать ответ в формате JSON и доставать из него нужное значение по ключу;
  • различать методы GET и POST и осознанно выбирать подходящий;
  • понимать, что такое API-ключ, и хранить его безопасно;
  • видеть в своей повседневной рутине задачи, которые можно автоматизировать через API, и описывать, какие запросы для этого нужны;
  • проверять, что ответ сервиса успешный, прежде чем брать из него данные.

3. Ключевые понятия

  • API — набор правил, по которым одна программа обращается к другой: что можно попросить и в каком виде придёт ответ [1].
  • REST API — самый распространённый вид API в интернете: обращение идёт по обычному веб-адресу (URL) и работает поверх протокола HTTP [1].
  • URL (адрес) — строка, указывающая, куда отправляется запрос.
  • HTTP-метод — слово, говорящее, что мы хотим сделать: GET (получить), POST (отправить/создать) и другие.
  • Параметры запроса — уточнения к запросу (например, город или дата).
  • JSON — текстовый формат данных в виде пар «ключ: значение», в котором чаще всего приходит ответ [3].
  • Ключ (поле) JSON — имя, по которому из ответа достают нужное значение.
  • API-ключ (токен) — секретный «пароль» твоей программы для доступа к сервису.
  • Код ответа (HTTP-статус) — число, показывающее, успешно ли выполнен запрос (например, 200 — успех, 404 — не найдено).
  • Эндпоинт (точка доступа) — конкретный адрес внутри API, отвечающий за одну операцию.

Полные определения собраны в глоссарии в конце материала.

4. Подробная теория

4.1. Что такое API: аналогия с кафе и меню

Начнём с главного определения, а потом разберём его по частям.

API (Application Programming Interface, «интерфейс программирования приложений») — это набор правил, по которым одна программа обращается к другой: что именно можно попросить и в каком виде придёт ответ [1].

Звучит абстрактно, поэтому держи в голове аналогию, которая объясняет суть лучше любого определения. Представь, что ты пришёл в кафе. Ты не идёшь на кухню, не зажигаешь плиту, не ищешь, где лежат продукты. Ты садишься за стол, берёшь меню, выбираешь блюдо и говоришь официанту: «мне, пожалуйста, вот это». Через время официант приносит готовое блюдо. Тебе не важно, как именно повар его готовил, какие у него ножи и сколько лет плите. Тебе важно одно: есть меню (список того, что можно заказать) и есть понятный способ сделать заказ и получить результат.

Вот это меню и есть API. Сервис (кафе) говорит тебе: «вот список того, что я умею отдавать, вот как нужно об этом просить, вот в каком виде ты получишь ответ». Ты (твоя программа) не знаешь и не должен знать, как сервис устроен внутри — какие у него базы данных, на каком языке он написан, где стоят его сервера. Ты знаешь только две вещи: что попросить и что ты получишь. Посмотри на схему «API как меню» (приложение А) — там запрос и ответ изображены как заказ официанту и принесённое блюдо.

Почему эта аналогия так важна? Потому что она объясняет главное преимущество API — скрытие сложности. Картографический сервис обрабатывает спутниковые снимки, строит маршруты, учитывает пробки — это гигантская система. Но тебе, чтобы показать карту в своём приложении, не нужно всё это уметь. Ты просто «делаешь заказ по меню»: просишь карту нужного города — и получаешь её. API превращает чужую огромную работу в простой запрос. Именно поэтому один разработчик в одиночку может собрать приложение, которое показывает погоду, карты, курсы валют и принимает оплату: всю тяжёлую работу делают чужие сервисы, а он лишь грамотно «заказывает» у них по их меню [2].

И ещё один важный вывод из аналогии: меню — это договор. Кафе обещает: если ты закажешь блюдо из меню так, как написано, ты получишь именно его. Так же и API: если ты составил запрос по правилам сервиса, ты получишь предсказуемый ответ. Нарушил правила (попросил то, чего в меню нет, или неправильно сформулировал) — получишь не блюдо, а сообщение об ошибке. Поэтому работа с API всегда начинается с чтения «меню» — документации сервиса, где описано, что и как можно у него просить.

4.2. REST API: API, который работает через веб-адрес

API бывают разные, но в интернете тебе чаще всего встретится один тип — REST API. Его особенность в том, что обращение к нему выглядит как обычный веб-адрес, а работает оно поверх того же протокола HTTP, по которому твой браузер открывает сайты [1].

Это очень удобно для понимания: ты уже пользуешься HTTP каждый день. Когда ты вводишь адрес сайта и жмёшь Enter, браузер отправляет на сервер запрос «дай мне эту страницу» и получает ответ — HTML-страницу. REST API работает точно так же, только вместо красивой страницы для человека он отдаёт «сырые» данные для программы — обычно в формате JSON. Можно сказать, что REST API — это сайт для программ, а не для людей.

Из этого следует приятная вещь: самый простой запрос к REST API можно сделать прямо в браузере. Если адрес API возвращает данные методом GET (о методах — ниже), ты просто вставляешь этот адрес в адресную строку, жмёшь Enter — и видишь ответ JSON прямо на экране. Это отличный способ потренироваться: не нужно ни кода, ни специальных программ. Позже, когда запросы станут сложнее, помогут инструменты вроде Postman или код на Python, но начать можно с браузера.

Запомни связку понятий: REST API → обращение по URL → поверх HTTP → ответ в JSON. Это каркас, на который ляжет всё остальное в теме.

4.3. Из чего состоит запрос: URL, метод, параметры

Любой запрос к REST API можно разобрать на части, как предложение по членам. Разберём на конкретном примере. Вот запрос за погодой:

GET https://api.weather.com/today?city=Almaty

В нём спрятаны три составные части. Посмотри на схему «API как меню» (приложение А) — там запрос как раз разложен на эти три блока.

1. URL (адрес) — куда обращаемся.
Это часть https://api.weather.com/today. Она говорит, к какому сервису и к какой именно его функции мы идём. Разберём её ещё мельче:

  • https:// — протокол. Буква «s» (secure) означает, что соединение защищённое (шифруется). Для API это важно — данные не должны идти по сети открытым текстом.
  • api.weather.com — адрес сервера (домен). Часто у API-сервисов он начинается со слова api, чтобы отличать его от обычного сайта для людей.
  • /today — путь к конкретной функции, это и есть эндпоинт (точка доступа). У одного сервиса может быть много эндпоинтов: /today — погода на сегодня, /forecast — прогноз, /history — архив. Каждый эндпоинт — это отдельная «строчка в меню».

2. Метод — что делаем.
В примере это слово GET в начале. Метод сообщает сервису намерение: хотим получить данные, отправить новые, изменить существующие или удалить. Самые частые — GET и POST, им посвящён следующий раздел. В адресной строке браузера метод не виден (браузер всегда использует GET), но в коде или в Postman он указывается явно.

3. Параметры — уточнения к запросу.
Это часть после знака вопроса: ?city=Almaty. Параметры уточняют, что именно мы хотим. Здесь мы говорим: погоду не вообще, а конкретно по городу Алматы. Правила записи параметров такие:

  • параметры начинаются со знака ?;
  • каждый параметр — это пара имя=значение (city=Almaty);
  • если параметров несколько, их разделяют знаком &, например: ?city=Almaty&day=tomorrow.

Соберём всё вместе. Запрос GET https://api.weather.com/today?city=Almaty читается так: «методом GET (получить данные) обращаюсь к сервису погоды, к функции "сегодня", уточняю — по городу Алматы». В ответ сервис пришлёт данные о сегодняшней погоде в Алматы.

Практический совет. Когда видишь незнакомый запрос, всегда раскладывай его на эти три части — так ты сразу понимаешь, что он делает. Сначала найди метод (что), потом базовый адрес и эндпоинт (куда), потом параметры после ? (с какими уточнениями). Этот разбор — половина умения работать с API.

4.4. Формат ответа JSON и как достать из него нужное поле

Запрос ушёл — сервис прислал ответ. Чаще всего ответ приходит в формате JSON (JavaScript Object Notation). Не пугайся названия: JSON — это просто текст, организованный по простым правилам, который удобно читать и человеку, и программе [3].

JSON строится из пар «ключ: значение». Вот ответ нашего сервиса погоды:

{
  "city": "Almaty",
  "temp": 18,
  "condition": "sunny"
}

Разберём правила записи:

  • весь объект заключён в фигурные скобки { };
  • внутри — пары "ключ": значение, разделённые запятыми;
  • ключ всегда в двойных кавычках: "temp";
  • значение может быть числом (18), строкой в кавычках ("Almaty"), логическим (true/false), а также вложенным объектом или списком.

Главное умение — достать значение по ключу. Это работает как поиск слова в словаре: ты знаешь ключ (temp) и по нему берёшь значение (18). В нашем примере:

  • по ключу "city" получаешь "Almaty";
  • по ключу "temp" получаешь 18;
  • по ключу "condition" получаешь "sunny".

На схеме «API как меню» (приложение А) этот шаг показан стрелкой «берёшь нужное поле по ключу: temp».

JSON бывает сложнее — со вложенностью. Например, сервис может вернуть так:

{
  "city": "Almaty",
  "weather": {
    "temp": 18,
    "wind": 3
  },
  "updated": "2026-06-26"
}

Здесь значение по ключу "weather" — это вложенный объект (ещё одни фигурные скобки). Чтобы взять температуру, идёшь по шагам: сначала берёшь weather, потом внутри него — temp. Получается путь weather → temp, значение 18. Думай об этом как о вложенных папках: чтобы добраться до файла, открываешь сначала внешнюю папку, потом внутреннюю.

А бывает, что сервис возвращает список (массив) — несколько однотипных объектов в квадратных скобках [ ]:

{
  "city": "Almaty",
  "forecast": [
    { "day": "today", "temp": 18 },
    { "day": "tomorrow", "temp": 21 }
  ]
}

Здесь forecast — список из двух элементов. Программа может пройти по нему циклом и взять температуру каждого дня. Списки появляются, когда данных «много однотипных»: список товаров, список городов, список записей открытых данных.

Алгоритм чтения любого ответа JSON:

  1. найди фигурные скобки { } — это границы объекта;
  2. прочитай ключи — это «названия полей», по ним понятно, что внутри;
  3. найди ключ, который тебе нужен;
  4. возьми его значение; если значение — вложенный объект { } или список [ ], спускайся внутрь по тому же правилу.

Это умение — фундамент всей работы с API: какой бы сервис ты ни взял, после запроса тебе всегда нужно «прочитать ответ и вытащить нужное поле».

4.5. GET и POST: два главных метода

Мы уже знаем, что метод говорит сервису, что мы хотим сделать. Двух методов хватает для огромной части задач — это GET и POST. Их разница принципиальна, и путать их нельзя. Посмотри на схему «GET и POST» (приложение Б) — там показано, чем они отличаются.

GET — получить данные.

  • Назначение: «просто посмотреть», прочитать информацию.
  • Ничего не меняет на сервере — сколько раз ни повтори запрос, данные на сервере останутся прежними. Такое свойство называют безопасным: GET не имеет побочных эффектов.
  • Параметры передаются прямо в адресе, после ? (как мы видели: ?city=Almaty).
  • Поэтому GET-запрос можно выполнить прямо в браузере.
  • Примеры: узнать погоду, посмотреть курс валют, получить список товаров, прочитать запись из открытых данных.

POST — отправить или создать данные.

  • Назначение: «что-то изменить или добавить» на сервере.
  • Меняет состояние сервера: после POST-запроса там появляется новая запись, отправляется сообщение, оформляется заказ. Повторишь дважды — может появиться две записи.
  • Данные передаются не в адресе, а в теле запроса (отдельной части, которую не видно в строке браузера). Это удобно и важно: в теле можно передать много данных, и они не светятся в адресе.
  • В браузере «просто так» POST не сделаешь — нужен код, Postman или форма на сайте.
  • Примеры: отправить сообщение в мессенджер, оформить заказ, зарегистрировать пользователя, добавить запись в базу.

Запомни простое правило выбора: если задача — «просто посмотреть данные», это GET. Если задача — «что-то изменить, создать или отправить», это POST. Спроси себя: «изменится ли что-то на сервере после моего запроса?» Нет — GET, да — POST.

Чтобы это закрепить, сведём различия в таблицу:

Признак GET POST
Зачем получить (посмотреть) данные отправить / создать данные
Меняет сервер? нет да
Где данные запроса в адресе после ? в теле запроса
Можно из браузера? да обычно нет
Пример узнать курс валют оформить заказ

Почему путать опасно. Если для операции «удалить заказ» ошибочно использовать GET, может получиться так, что простой переход по ссылке или повтор запроса случайно удалит данные — ведь GET считается «безобидным просмотром», его могут автоматически повторять браузеры и поисковики. И наоборот, если безобидный просмотр оформить как POST, ты усложнишь себе жизнь без причины. Поэтому метод выбирают по смыслу действия, а не наугад.

Кроме GET и POST существуют и другие методы — например, PUT (заменить данные) и DELETE (удалить). Их логика та же: метод выражает намерение. Но для начала тебе хватит твёрдо различать GET и POST.

4.6. API-ключ и безопасное хранение секретов

Многие полезные API требуют, чтобы ты «представился» — доказал, что имеешь право пользоваться сервисом. Для этого служит API-ключ (его ещё называют токеном) — длинная строка вроде a7f3c9e1b4..., которую сервис выдаёт тебе при регистрации.

Самое важное, что нужно усвоить: API-ключ — это пароль твоей программы к сервису. Кто знает ключ, тот может обращаться к сервису от твоего имени. Отсюда — главная и очень частая ошибка новичков: написать ключ прямо в коде и выложить этот код в открытый доступ, например, в публичный репозиторий на GitHub.

Чем это опасно? Тем, что открытые репозитории постоянно прочёсывают автоматические сканеры — программы, которые ищут в чужом коде что-то похожее на ключи. Найденный ключ используют в считанные минуты: за твой счёт расходуют платные лимиты сервиса, крадут доступные через сервис данные, а тебя могут заблокировать за «подозрительную активность», которую ты даже не совершал. И «я выложил всего на час» не спасает — сканеры работают круглосуточно.

Как хранить ключ правильно:

  • Не пиши ключ прямо в коде. Код видят коллеги, он попадает в репозиторий, его легко случайно опубликовать.
  • Храни ключ отдельно от кода — в переменных окружения (специальных настройках, которые программа читает при запуске) или в защищённом хранилище. Тогда в самом коде вместо ключа стоит лишь ссылка «возьми ключ из окружения».
  • Не публикуй файлы с ключами. Файл, где лежат секреты, добавляют в список «не выкладывать» (в Git это .gitignore), чтобы он не попал в репозиторий.
  • При утечке — сразу отзови ключ. Если ключ всё же засветился, не надейся, что «никто не заметит». Зайди в сервис, отзови (аннулируй) старый ключ и получи новый. Старый после этого перестанет работать — и украденная копия станет бесполезной.

Этот принцип шире, чем API: пароли, ключи и любые секреты никогда не хранят в коде и не выкладывают в открытый доступ. Привычка отделять секреты от кода — признак аккуратного разработчика. Здесь же она прямо связана с законом: если по украденному ключу утекут чужие персональные данные, это уже нарушение Закона РК «О персональных данных и их защите» № 94-V [4], а вопросы защиты информации в целом регулирует Закон РК «Об информатизации» № 418-V [5].

4.7. Проверка успешности ответа: код состояния

Запрос не всегда срабатывает: можно ошибиться в адресе, сервис может быть недоступен, ключ может оказаться неверным. Поэтому прежде чем брать данные из ответа, нужно убедиться, что запрос вообще удался. Для этого служит код состояния (HTTP-статус) — число, которое сервис присылает вместе с ответом.

Коды сгруппированы по первой цифре, и достаточно запомнить логику групп:

  • 2xx — успех. Самый частый — 200 OK: всё хорошо, данные пришли. Можно брать поля из JSON.
  • 4xx — ошибка на твоей стороне. Например, 404 Not Found — такого адреса/ресурса нет; 401 Unauthorized — не представился (нет ключа или он неверный); 400 Bad Request — запрос составлен неправильно.
  • 5xx — ошибка на стороне сервера. Например, 500 — у сервиса что-то сломалось. Твой запрос мог быть верным, но сервис не справился.

Практический вывод: сначала смотри код, потом данные. Алгоритм такой: получил ответ → проверил, что статус из группы 2xx → только тогда читаешь JSON и достаёшь поля. Если код 4xx — ищи ошибку у себя (адрес, параметры, ключ). Если 5xx — проблема у сервиса, имеет смысл повторить позже. Если этот шаг пропустить, программа попытается «достать поле» из ответа, в котором вместо данных лежит сообщение об ошибке, — и сломается.

4.8. Где это применяют: автоматизация рутины

Теперь, когда ты понимаешь запрос, ответ, методы и ключ, соберём картину применения. API нужны там, где программы должны обмениваться данными, и особенно — где человек иначе делал бы это руками. Три типовых направления:

  • Получение данных (GET). Погода, курсы валют, открытые государственные данные. В Казахстане для этого есть портал data.egov.kz — там опубликованы открытые наборы данных с доступом по API [9]. Вместо того чтобы вручную копировать таблицу с сайта, программа делает GET-запрос и получает свежие данные сама.
  • Отправка данных (POST). Сообщение в мессенджер, запись в таблицу или базу, отправка формы. Например, бот, который каждое утро сам отправляет тебе расписание, использует POST к API мессенджера.
  • Связка сервисов. Самое мощное применение: соединить несколько API в цепочку. Например, программа делает GET к сервису курсов, берёт число и делает POST в твою таблицу. Так из отдельных «кирпичей» собирается автоматизация, которая работает без человека.

Главная идея автоматизации: ты тратишь время один раз — настраиваешь запросы, — а дальше работа выполняется сама, каждый день, без ошибок и без твоего участия. Это превращает повторяющуюся ручную рутину в фоновый процесс. Вернись к примеру из введения с курсом валют: один раз настроил GET-запрос и запись в таблицу — и больше никогда не переписываешь число руками.

ИИ-ассистент как помощник. Здесь очень кстати приходит ИИ. Ты можешь описать задачу словами — «сделай запрос к такому-то API и достань поле temp» — и попросить готовый пример запроса или короткий скрипт. Но помни принцип ответственной работы с ИИ из занятия 01: ИИ может ошибиться (придумать несуществующий эндпоинт или неверный ключ JSON), поэтому его пример нужно прочитать, проверить по документации сервиса и протестировать, а не вставлять вслепую [6]. ИИ ускоряет черновик, но отвечаешь за результат ты.

4.9. Пошаговый алгоритм работы с новым API

Чтобы не теряться перед незнакомым сервисом, держи универсальный порядок действий:

  1. Найди документацию сервиса (его «меню»). Там описаны эндпоинты, методы, параметры и формат ответа.
  2. Пойми, что тебе нужно, и подбери эндпоинт. Реши, какой метод подходит: смотришь данные — GET, меняешь — POST.
  3. Собери запрос: базовый URL + эндпоинт + нужные параметры после ?.
  4. Узнай, нужен ли ключ. Если да — зарегистрируйся, получи ключ и сразу подумай, где его безопасно хранить (не в коде!).
  5. Выполни запрос — для GET без ключа можно прямо в браузере, иначе через Postman или код.
  6. Проверь код состояния. 2xx — успех, идём дальше; 4xx/5xx — разбираемся с ошибкой.
  7. Прочитай JSON и достань нужные поля по ключам (для вложенных — спускайся внутрь по шагам).
  8. Используй данные — выведи, запиши, передай дальше. Если автоматизируешь — настрой повтор по расписанию.

Этот алгоритм работает с любым REST API, от погоды до открытых данных РК.

5. Разобранные примеры и мини-кейсы

Пример 1 — разбор GET-запроса по частям.
Дан запрос:

GET https://api.weather.com/today?city=Almaty

Разбор:

  • метод — GET: получаем данные, на сервере ничего не меняем;
  • URL — https://api.weather.com/today: сервис погоды (api.weather.com), эндпоинт «сегодня» (/today), защищённое соединение (https);
  • параметр — city=Almaty: уточняем город.

Что вернёт: JSON с погодой по Алматы на сегодня. Поскольку это GET без ключа, такой запрос можно вставить в адресную строку браузера и сразу увидеть ответ.

Пример 2 — чтение ответа JSON.
Сервис вернул:

{ "city": "Almaty", "temp": 18, "condition": "sunny" }

Разбор: чтобы взять температуру, обращаемся по ключу temp → получаем 18. Чтобы взять погодное условие, обращаемся по ключу condition → получаем "sunny". Город узнаём по ключу city → "Almaty". Если бы программе нужно было показать строку «В Алматы +18, солнечно», она собрала бы её из этих трёх полей.

Пример 3 — GET или POST.
Две задачи: (а) получить список товаров магазина; (б) добавить в магазин новый заказ.
Разбор: (а) — мы смотрим данные, на сервере ничего не меняется → метод GET. (б) — мы создаём новую запись (заказ), состояние сервера меняется → метод POST, данные заказа уйдут в теле запроса. Контрольный вопрос «изменится ли что-то на сервере?» даёт ответ сразу: в (а) нет, в (б) да.

Пример 4 — мини-кейс автоматизации курса валют.
Задача: каждое утро записывать курс доллара в таблицу.
Как было вручную: заходишь на сайт банка, находишь курс, переписываешь в таблицу — каждый день, с риском забыть или ошибиться.
Как через API: один раз настраиваешь программу так, чтобы она по расписанию делала GET-запрос к сервису курсов, проверяла код ответа (200), брала из JSON поле с курсом по ключу и делала запись в таблицу. Дальше всё работает само. Разбор: ручная рутина (10 секунд × каждый день) превратилась в разовую настройку и надёжный автоматический процесс. Это и есть смысл автоматизации через API.

6. Частые ошибки и как их избежать

  • Ошибка: хранить API-ключ прямо в коде и выкладывать его на GitHub. Почему: ключ — пароль программы; его находят автоматические сканеры и используют от твоего имени (чужой расход, утечка, блокировка). Как правильно: хранить ключ отдельно (переменные окружения), не публиковать, при утечке сразу отзывать и менять [4].
  • Ошибка: брать поля из ответа, не проверив код состояния. Почему: в ответе может лежать сообщение об ошибке (404, 401), а не данные, и программа сломается. Как правильно: сначала проверь, что статус из группы 2xx, потом читай JSON.
  • Ошибка: путать GET и POST. Почему: GET для изменения данных опасен (его могут случайно повторить), а POST для простого просмотра — лишнее усложнение. Как правильно: «посмотреть» — GET, «изменить/создать» — POST; спроси себя, меняется ли что-то на сервере.
  • Ошибка: искать значение по неверному ключу или забыть про вложенность. Почему: ключи чувствительны к точному написанию, а нужное поле может лежать внутри другого объекта. Как правильно: сверь ключ с документацией; для вложенного JSON спускайся по шагам (weather → temp).
  • Ошибка: доверять примеру запроса от ИИ без проверки. Почему: ИИ может выдумать несуществующий эндпоинт или неверный ключ. Как правильно: сверь с документацией сервиса и протестируй запрос [6].

7. Памятка: «Прежде чем отправить запрос»

Короткий чек-лист для работы с любым API:
  1. нашёл документацию (меню) сервиса;
  2. выбрал метод по смыслу: смотрю — GET, меняю — POST;
  3. собрал запрос: URL + эндпоинт + параметры после ?;
  4. если нужен ключ — храню его отдельно от кода, не публикую;
  5. выполнил запрос и проверил код состояния (нужен 2xx);
  6. прочитал JSON и достал нужные поля по ключу;
  7. использовал данные; если автоматизирую — настроил повтор.

8. Краткие итоги

  • Воспринимай API как «меню» запросов к чужой программе: ты знаешь, что попросить и что получишь, не зная устройства внутри.
  • В интернете чаще всего встречается REST API — «сайт для программ»: обращение по URL поверх HTTP, ответ в JSON.
  • Запрос состоит из трёх частей: URL (куда) + метод (что) + параметры (уточнения после ?).
  • Ответ обычно в JSON — пары «ключ: значение»; нужное берёшь по ключу, для вложенного спускаешься внутрь.
  • GET — получить (ничего не меняет), POST — отправить/создать (меняет сервер, данные в теле запроса).
  • API-ключ — это пароль программы; храни его отдельно от кода и никогда не публикуй.
  • Перед чтением данных проверь код состояния: нужен успех из группы 2xx.
  • Главный смысл API для тебя — автоматизация: один раз настроил запросы, дальше машина делает рутину сама.

9. Вопросы для самопроверки

  1. Объясни на аналогии «кафе и меню», что такое API и почему тебе не нужно знать устройство сервиса внутри.
  2. Чем REST API похож на обычный сайт и чем отличается? Почему простой GET-запрос можно сделать прямо в браузере?
  3. Разбери запрос GET https://api.weather.com/today?city=Almaty на три части. Что означает каждая?
  4. Дан ответ {"city":"Almaty","temp":18,"condition":"sunny"}. По каким ключам взять температуру и погодное условие и какие значения получишь?
  5. Чем GET отличается от POST? Приведи по одному примеру задачи для каждого и объясни выбор.
  6. Что такое API-ключ и почему его нельзя писать прямо в коде и выкладывать на GitHub? Что делать при утечке?
  7. Зачем проверять код состояния ответа до того, как брать из него данные? Что означают группы 2xx, 4xx, 5xx?
  8. Опиши своими словами, какую ежедневную рутину можно автоматизировать через API и какие запросы для этого понадобятся.

10. Глоссарий

  • API — набор правил, по которым одна программа обращается к другой: что можно попросить и в каком виде придёт ответ.
  • API-ключ (токен) — секретная строка-«пароль» программы для доступа к сервису; хранится отдельно от кода.
  • GET — HTTP-метод для получения данных; ничего не меняет на сервере, параметры идут в адресе.
  • HTTP — протокол, по которому браузеры и программы обмениваются запросами и ответами с серверами.
  • JSON — текстовый формат данных в виде пар «ключ: значение»; основной формат ответа API.
  • Код состояния (HTTP-статус) — число об успешности запроса: 2xx — успех, 4xx — ошибка клиента, 5xx — ошибка сервера.
  • Ключ (поле) JSON — имя, по которому из ответа достают значение.
  • Метод (HTTP-метод) — слово, выражающее намерение запроса (GET, POST, PUT, DELETE).
  • Параметры запроса — уточнения после знака ? в адресе, в виде пар имя=значение.
  • POST — HTTP-метод для отправки/создания данных; меняет состояние сервера, данные идут в теле запроса.
  • REST API — распространённый вид API: обращение по URL поверх HTTP, ответ обычно в JSON.
  • Тело запроса — часть POST-запроса, в которой передаются отправляемые данные (не видна в адресе).
  • URL (адрес) — строка, указывающая, куда отправляется запрос.
  • Эндпоинт (точка доступа) — конкретный адрес внутри API, отвечающий за одну операцию.
