# Техническая документация в текстовом редакторе

Код без документации — это «работает, но как — знаю только я». README, инструкция по установке, описание API, техзадание — всё это пишет разработчик. И пишет не «красиво», а **структурно**: чтобы другой человек (или ты через полгода) понял за минуту. Этот урок — про то, как оформлять техническую документацию в текстовом редакторе так, чтобы ею пользовались.

## Чему ты научишься

- отличать техническую документацию от «текста ради текста»;
- структурировать документ заголовками, списками и таблицами;
- оформлять README и инструкцию по понятной схеме.

## Документация — это инструмент, а не формальность

Хорошая документация отвечает читателю на вопросы: *что это, как запустить, как пользоваться, что делать при ошибке.* Если документ не помогает действовать — он бесполезен.

## Структура важнее оформления

Инструменты (MS Word, Google Docs, или Markdown в редакторе кода) дают одно и то же: **заголовки, списки, таблицы, выделение, код-блоки.** Главное — иерархия:

- **Заголовки** (Заголовок 1 / 2 / 3) — каркас документа, по ним строится оглавление.
- **Списки** — шаги и перечни (нумерованные для последовательности, маркированные для набора).
- **Таблицы** — для сравнения (параметр → значение).
- **Моноширинный/код** — для команд, имён файлов, кода.

### Схема README проекта
```
# Название проекта
Краткое описание (1–2 предложения)
## Установка
шаги по пунктам
## Использование
пример запуска
## Возможные ошибки
проблема → решение
```

### Мини-кейс
Студент сдал проект с README из одной строки «Запускается через main». Преподаватель не смог запустить. Причина: нет шагов установки и команды запуска. Следующий шаг: добавить раздел «Установка» с командами по пунктам.

## Частые ошибки

**Ошибка:** оформлять заголовки жирным текстом вручную вместо стилей «Заголовок».
**Почему:** не строится оглавление, ломается навигация и структура.
**Как правильно:** использовать стили заголовков (или `#` в Markdown).

**Ошибка:** «простыня» текста без списков и разбивки.
**Почему:** инструкцию невозможно выполнять по шагам.
**Как правильно:** шаги — нумерованным списком, команды — в код-блоке.

## Итог: что теперь делать

- Пиши документацию под действие: что, как запустить, как пользоваться, что при ошибке.
- Используй стили заголовков, а не ручное форматирование.
- Команды и имена файлов — моноширинным/код-блоком.
- README — по схеме: описание → установка → использование → ошибки.
