Техническая документация в текстовом редакторе

Код без документации — это «работает, но как — знаю только я». README, инструкция по установке, описание API, техзадание — всё это пишет разработчик. И пишет не «красиво», а структурно: чтобы другой человек (или ты через полгода) понял за минуту.

Чему ты научишься: отличать техдокументацию от «текста ради текста»; структурировать документ заголовками, списками и таблицами; оформлять README по понятной схеме.

Документация — инструмент, а не формальность

Хорошая документация отвечает на вопросы: что это, как запустить, как пользоваться, что делать при ошибке. Если документ не помогает действовать — он бесполезен.

Структура важнее оформления

Любой инструмент (Word, Google Docs, Markdown) даёт одно и то же. Главное — иерархия:

  • Заголовки (1/2/3) — каркас, по ним строится оглавление.
  • Списки — шаги (нумерованные) и перечни (маркированные).
  • Таблицы — для сравнения (параметр → значение).
  • Код-блок — для команд, имён файлов, кода.

Схема README проекта

# Название проекта
Краткое описание (1–2 предложения)
## Установка
шаги по пунктам
## Использование
пример запуска
## Возможные ошибки
проблема → решение

Мини-кейс

Студент сдал README из одной строки «Запускается через main». Преподаватель не смог запустить. Причина: нет шагов установки и команды запуска. Решение: добавить раздел «Установка» с командами по пунктам.

Ошибка: заголовки оформлены жирным вручную вместо стилей «Заголовок».
Почему: не строится оглавление, ломается структура.
Как правильно: использовать стили заголовков (или # в Markdown).
Ошибка: «простыня» текста без списков.
Почему: инструкцию нельзя выполнять по шагам.
Как правильно: шаги — нумерованным списком, команды — код-блоком.
Итог — что теперь делать:
  • пиши документацию под действие: что, как запустить, как пользоваться, что при ошибке;
  • используй стили заголовков, а не ручное форматирование;
  • команды и имена файлов — код-блоком;
  • README по схеме: описание → установка → использование → ошибки.
