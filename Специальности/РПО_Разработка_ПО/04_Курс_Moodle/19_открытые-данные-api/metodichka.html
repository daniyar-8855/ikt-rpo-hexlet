Занятие 19. Использовать порталы открытых данных (data.egov.kz) и их API

Дисциплина: Применение информационно-коммуникационных и цифровых технологий
Результат обучения: РО 2.2 — использование услуг веб-порталов и цифровых сервисов · Объём темы: 2 часа
Квалификация: 4S06130103 «Разработчик программного обеспечения»

Это расширенный разбор темы для самостоятельного изучения. Он подробнее, чем конспект урока: здесь мы не просто говорим «есть портал и есть API», а идём по реальному пути — заходим на data.egov.kz, скачиваем настоящий набор, открываем скачанный JSON и разбираем его строка за строкой, дёргаем реальный API-запрос и читаем, что вернул сервер. С ловушками, на которых спотыкаются почти все, кто делает это в первый раз. Читай не как справочник, а как рассказ практика: лучше всего — с открытым браузером рядом, чтобы сразу повторять руками. Данные осваиваются на настоящих данных, а не на выдуманных.

1. Введение: зачем тебе эта тема

Расскажу, как это выглядит на защите, потому что я видел эту сцену не раз. Студент делает дашборд «Население городов Казахстана по годам». Дедлайн близко, лезть куда-то за данными лень, и он делает проще всего: вписывает цифры «на глаз». Алматы — «примерно два миллиона», Шымкент — «около миллиона», рисует красивые столбики. Внешне солидно, картинка защиты не стыдная. А потом один вопрос от комиссии — «Откуда числа?» — и всё рассыпается. Числа выдуманы, значит выводы недостоверны, значит вся работа — картинка ради картинки. И оценка соответствующая.

А ведь рядом лежал другой путь, и он был не сложнее. Государство уже собрало эту статистику, проверило её и бесплатно выложило в открытый доступ — заходи и бери. Это и есть открытые данные. Открываешь портал data.egov.kz, за пять минут находишь набор по населению, скачиваешь — и строишь тот же дашборд, но на проверенных числах. И теперь на вопрос «откуда?» у тебя готовый ответ: вот название набора, вот орган, который его опубликовал, вот прямая ссылка. Разница между этими двумя студентами не в уме и не в дизайне — она ровно в одном: один знал, где лежат настоящие данные, а второй нет.

Вторая половина темы — про случай, когда данные меняются постоянно. Курс валют, прогноз погоды, расписание транспорта, число свободных мест — качать такой файл руками каждый час бессмысленно, он устаревает быстрее, чем ты его открыл. Для этого есть API: твоя программа сама шлёт запрос сервису и получает свежий ответ, обычно в формате JSON. Так работает почти всё, чем ты пользуешься каждый день. Погодный виджет на телефоне не хранит погоду — он дёргает API метеосервиса. Конвертер валют не знает курс сам — он спрашивает его у чужого API. Лента новостей, карта в приложении такси, баланс в банковском приложении — под капотом везде один и тот же приём: запрос ушёл, ответ пришёл, показали тебе результат.

Эта тема — фундамент работы с данными, а данные сегодня есть в любом проекте. Дашборд, прототип, учебный анализ, обучение модели машинного обучения — всё начинается с одного и того же вопроса: «где взять настоящие данные?». Разработчик, который умеет находить открытые наборы и понимает, как устроен обмен через API, не тратит время на выдумывание и не выглядит несерьёзно на защите или на собеседовании. Этот навык напрямую относится к РО 2.2 — умению находить, оценивать и использовать информацию из цифровых источников [1; 8].

2. Что ты узнаешь и чему научишься

После изучения темы ты сможешь:

  • объяснять, что такое открытые данные и чем они отличаются от персональных;
  • находить и скачивать наборы открытых данных на портале data.egov.kz;
  • различать форматы CSV, JSON и XLSX и осознанно выбирать подходящий под задачу;
  • объяснять, что такое API и зачем он нужен разработчику;
  • описывать схему обмена «запрос → ответ JSON» и читать простой JSON-ответ;
  • собрать и прочитать реальный GET-запрос к API портала data.egov.kz;
  • понимать, почему открытые данные всегда обезличены, и не путать их с персональными [3];
  • решать, когда достаточно один раз скачать файл, а когда нужно получать данные через API.

3. Ключевые понятия

  • Открытые данные — наборы данных, опубликованные свободно и бесплатно для повторного использования; они обезличены и доступны любому [1].
  • Набор данных (датасет) — структурированная коллекция данных по одной теме (например, население городов по годам), обычно в виде таблицы или файла.
  • Портал открытых данных — государственный или организационный сайт, где публикуются и каталогизируются наборы данных. В Казахстане это data.egov.kz.
  • Формат данных — способ записи данных в файле. Основные: CSV, JSON, XLSX.
  • API (Application Programming Interface) — программный интерфейс: способ одной программы запрашивать данные или функции у другой.
  • Запрос (request) — обращение программы к API: «дай мне такие-то данные».
  • Ответ (response) — то, что API возвращает в ответ, обычно в формате JSON.
  • JSON — текстовый формат записи данных в виде пар «ключ–значение», удобный для программ.
  • Обезличивание — удаление из данных признаков, по которым можно опознать конкретного человека.
  • Персональные данные — сведения, по которым можно прямо или косвенно опознать человека (ФИО, ИИН, телефон) [3].

4. Подробная теория

4.1. Что такое открытые данные и почему они вообще существуют

Начнём с главного понятия темы.

Открытые данные — это наборы данных, которые государство или организация опубликовали свободно и бесплатно, чтобы любой человек мог их скачать, использовать и распространять без ограничений. Такие данные обезличены — по ним нельзя опознать конкретного человека.

Разберём это определение по частям, потому что в каждом слове зашит смысл, и на защите тебя могут спросить именно про эти слова.

«Свободно и бесплатно». Тебе не нужно ни у кого спрашивать разрешения, платить или заключать договор, чтобы взять набор. Открытые данные для того и публикуют, чтобы их брали и использовали — в проектах, исследованиях, приложениях, журналистике. Это прямая противоположность закрытым данным, к которым доступ ограничен паролем, договором или деньгами. Оговорюсь честно: «бесплатно скачать» и «делать с этим что угодно» — не всегда одно и то же, у набора может быть указана лицензия и условия ссылки на источник. Но для учебного проекта тебе достаточно правила: взял — сошлись на источник.

«Повторное использование». Ключевая идея в том, что одни и те же данные пригождаются многим разным людям для совершенно разных задач. Государство собрало статистику по школам один раз — а работает она на всех: студент строит из неё дашборд, журналист пишет статью, стартап делает сервис поиска школ. Собрали один раз — используют многократно. По-английски это называют принципом «collect once, use many times», и он объясняет, почему открытые данные вообще выгодны государству: не надо отвечать каждому по отдельности, выложил один раз — и все берут сами.

«Обезличены». Вот это — самое важное для понимания границы, и именно здесь новички ошибаются. Открытые данные — это всегда агрегированные (обобщённые) или обезличенные сведения: «в городе N столько-то жителей», «в области M столько-то школ». По таким данным невозможно сказать, кто конкретно живёт в городе N. А вот список с ФИО, ИИН и телефонами реальных людей открытыми данными быть не может — это персональные данные, их публикация запрещена законом [3]. Запомни простое правило, оно спасёт тебя и на экзамене, и потом на работе: открытые данные отвечают на вопрос «сколько / какой в среднем», но никогда — «кто именно».

Посмотри на схему «Что такое открытые данные» (приложение А). Она укладывает всю логику в три блока. Слева — государство, которое публикует данные. В центре — набор данных на портале data.egov.kz в форматах XLSX, JSON или XML. Справа — любой пользователь, который берёт набор и строит на нём дашборды и проекты. Стрелки идут слева направо: данные движутся от того, кто их собрал, к тому, кто применяет. А внизу схемы — главное ограничение: данные обезличены и публичны, поэтому их можно свободно скачивать, но персональные данные так публиковать нельзя.

Зачем это нужно государству. Помимо экономии, тут есть и общественный смысл. Когда государство открывает статистику, оно становится прозрачнее: любой может проверить цифры, построить на них сервисы, заметить проблему. Это часть политики цифрового государства и государственной программы цифровизации, которую в Казахстане поддерживает Закон РК «Об информатизации» № 418-V [2].

Зачем это нужно лично тебе как разработчику. Ещё проще: тебе постоянно нужны настоящие данные. Без них не построишь честный дашборд, не сделаешь реалистичный прототип, не обучишь модель. Выдумывать данные — значит с самого начала делать недостоверный продукт, который развалится от первого вопроса. Открытые данные дают тебе реальную фактуру легально и бесплатно. Это твой первый и самый доступный источник данных в любом проекте — с него и начинай.

4.2. Портал data.egov.kz: где и как искать наборы

Главный портал открытых данных Казахстана — data.egov.kz. Это часть инфраструктуры электронного правительства (eGov). Здесь государственные органы публикуют тысячи наборов данных: статистику по регионам, справочники, экономические и социальные показатели.

Что там реально можно найти. Тематика широкая, и почти под любой учебный проект набор найдётся:

  • демография — население по городам и областям, рождаемость, по годам;
  • образование — число школ, колледжей, вузов, количество студентов по регионам;
  • здравоохранение — медицинские учреждения, показатели по областям;
  • экономика и занятость — зарплаты, безработица, отрасли;
  • цифровизация и связь — доступ к интернету, число абонентов;
  • транспорт и ЖКХ — дороги, перевозки, инфраструктура.

Эти направления не случайны — именно по ним построены варианты задания к этому занятию (В1–В6). Так что, выбирая тему для дашборда, ты сразу попадаешь в реальный раздел портала, а не блуждаешь наугад.

Как искать набор — пошагово, как я делаю это сам. Чтобы не тонуть в каталоге, действуй по шагам:

  1. Открой data.egov.kz в браузере.
  2. Найди строку поиска или каталог наборов данных (datasets). Вводи тему своими словами: «население», «школы», «занятость». Не мудри с формулировками — портал ищет по обычным словам.
  3. Открой подходящий набор и прочитай его карточку. Это тот шаг, который новички проскакивают и потом жалеют. В карточке написано: название, описание, какой орган опубликовал данные, за какой период, в каких форматах доступна выгрузка и когда набор в последний раз обновлялся.
  4. Оцени, подходит ли набор: тот ли это период? есть ли нужные тебе колонки — например, и город, и год, и численность одновременно? в удобном ли формате выгрузка? Бывает, что набор вроде по теме, а внутри только одна область или только один год — вот это и ловишь на карточке, до скачивания.
  5. Скачай файл в нужном формате (кнопка «Скачать» / «Экспорт»; портал предлагает XLSX, JSON или XML).

Что обязательно зафиксировать сразу. Как только берёшь набор для проекта, тут же выпиши его «паспорт»: точное название, формат, период данных и прямую ссылку. Не «потом», а сразу — потому что «потом» ты этот набор среди тысяч других уже не найдёшь. Нужно это по двум причинам. Во-первых, на защите ты обязан показать источник — это и есть доказательство, что данные настоящие. Во-вторых, без ссылки ты сам через неделю не вспомнишь, откуда взял цифры, и не сможешь их обновить. В задании к занятию указание реквизитов и ссылки прямо оценивается баллами — это не бюрократия, а профессиональная привычка ссылаться на источник, которую с тебя будут спрашивать всю карьеру.

Оценка достоверности — не отключай голову. Даже на официальном портале данные бывают старыми или дырявыми. Смотри на дату обновления (не устарел ли набор), на полноту (нет ли пропущенных лет и пустых ячеек), на то, какой орган опубликовал. Если видишь, что последний год в наборе — позапрошлый, а тебе нужен свежий, это повод искать другой набор, а не молча тащить устаревшие цифры в проект. Этот навык — критическая оценка цифровых источников — относится к РО 2.2 и подробно разбирается в занятии 11; здесь ты просто применяешь его на практике [8].

4.3. Форматы данных: CSV, JSON и XLSX

Сразу важная и точная деталь, которую надо держать в голове: сам портал data.egov.kz отдаёт наборы в форматах XLSX, JSON и XML. CSV в списке форматов портала нет — но CSV настолько распространён у других источников данных, что не разобрать его нельзя. Поэтому дальше мы смотрим три самых ходовых формата вообще — CSV, JSON и XLSX. Различать их нужно потому, что от формата напрямую зависит, сможешь ли ты удобно обработать данные кодом или только посмотреть глазами.

ФорматЧто этоДля кого удобен
CSVтаблица, где значения в строке разделены запятымидля кода и для Excel — универсальный
JSONструктура из пар «ключ–значение»для API и программ
XLSXфайл таблицы Excel со стилями и формуламидля людей, для просмотра

Разберём каждый вживую, на одних и тех же данных — населении городов.

CSV (Comma-Separated Values — «значения, разделённые запятыми»). Самый простой табличный формат. Обычный текстовый файл: первая строка — заголовки колонок, каждая следующая строка — одна запись, значения внутри строки разделены запятыми (в русской локали Excel нередко ставит вместо запятой точку с запятой — это классическая причина, почему «CSV открылся одной колонкой», имей в виду). Вот как выглядит набор «население городов» в CSV:

city,year,population
Алматы,2020,1916779
Алматы,2021,1977011
Шымкент,2020,1038155
Шымкент,2021,1074880

Чем хорош CSV: его понимает почти всё — Excel, Google Таблицы, Python, любой язык. Файл маленький, читается и человеком, и программой. Именно поэтому CSV — рабочая лошадка анализа данных. Чем плох: он плоский, в него нельзя удобно уложить вложенную структуру (таблицу внутри таблицы) — а как только данные становятся сложнее простой сетки, CSV начинает трещать.

JSON (JavaScript Object Notation). Текстовый формат записи данных в виде пар «ключ–значение». Он умеет описывать не только плоские таблицы, но и вложенные структуры, поэтому стал стандартом обмена между программами и основным форматом ответов API. Те же данные о населении в JSON:

[
  { "city": "Алматы",  "year": 2020, "population": 1916779 },
  { "city": "Алматы",  "year": 2021, "population": 1977011 },
  { "city": "Шымкент", "year": 2020, "population": 1038155 }
]

Читается так: квадратные скобки [ ] — это список (массив) записей; фигурные скобки { } — одна запись (объект); внутри пары "ключ": значение. То есть «город — Алматы, год — 2020, население — 1916779». Программе такую структуру разобрать легко: она берёт значение по имени ключа (record["population"]), а не считает запятые вручную, как в CSV, и не боится, что порядок колонок поменялся. Одна честная ловушка на будущее: в JSON важны кавычки и запятые — лишняя запятая после последнего элемента или одинарные кавычки вместо двойных, и парсер ругнётся. Подробное описание формата с примерами есть в документации MDN [5; 7]. JSON ты будешь встречать постоянно — как только начнёшь работать с API, он станет твоим ежедневным чтением.

XLSX (формат Microsoft Excel). «Родной» формат Excel: таблица со стилями, цветами, объединёнными ячейками, несколькими листами и формулами. Отлично подходит, чтобы человек открыл файл и посмотрел данные глазами. Но для кода он неудобен: внутри это, по сути, сжатый архив из XML-файлов, а всё оформление — объединённые ячейки, шапки в две строки, пустые строки «для красоты» — мешает автоматической обработке. Скажу честно из опыта: чаще всего боль при работе с XLSX не в самом формате, а в том, что люди делают таблицу «для глаз» — с объединёнными ячейками и заголовком через весь лист, — а потом её приходится читать кодом. Поэтому: планируешь обрабатывать данные программой — бери CSV или JSON; данные нужны только посмотреть руками — подойдёт XLSX.

Простое правило выбора. Запомни так: CSV и JSON — для кода и автоматической обработки; XLSX — для глаз человека. А между CSV и JSON выбор такой: CSV — когда данные плоские, как таблица; JSON — когда есть вложенность или когда данные приходят из API (а из API они почти всегда приходят именно в JSON).

4.4. Что такое API и зачем он нужен

Скачать файл руками — нормальный вариант, когда данные меняются редко. Но что делать, если они меняются каждую минуту? Курс валют, погода, цена акции, число свободных мест — качать файл заново вручную физически невозможно, он устареет раньше, чем ты дважды кликнешь. Здесь и нужен API.

API (Application Programming Interface — «интерфейс программирования приложений») — это способ одной программы запрашивать данные или функции у другой. Вместо ручного скачивания файла программа делает запрос и получает ответ, обычно в формате JSON.

Понять API проще всего через аналогию с рестораном. Ты (твоя программа) сидишь за столом и хочешь блюдо (данные). Ты не идёшь сам на кухню (в чужую базу данных) — это не твоя зона, и тебя туда никто не пустит. Вместо этого ты делаешь заказ официанту (API). Официант передаёт заказ на кухню, забирает готовое блюдо и приносит тебе. Тебе не надо знать, как устроена кухня — надо лишь знать, что можно заказать и как правильно попросить. API — это официант между твоей программой и чужими данными. Он прячет внутреннюю кухню и даёт удобный, понятный способ получить результат.

Зачем это разработчику:

  • актуальность — данные приходят свежими в момент запроса, без ручного обновления;
  • автоматизация — программа сама получает и обновляет данные по расписанию или по событию;
  • интеграция — твоё приложение может использовать чужие сервисы (карты, оплату, погоду), не реализуя их с нуля.

Почти всё, чем ты пользуешься, работает через API. Погодный виджет показал температуру — он запросил её у API метеосервиса. Приложение банка показало курс доллара — оно дёрнуло API. Сайт встроил карту — обратился к API карт. Умение работать с API — базовый навык интеграций, и именно с него начинаются темы автоматизации дальше в курсе (занятия 34–35) [4; 6].

4.5. Как идёт обмен: «запрос → ответ JSON»

Теперь разберём сам обмен пошагово. Посмотри на схему «Обращение к открытым данным через API» (приложение Б): слева — твоя программа, справа — сервис (API), между ними две стрелки. Верхняя (синяя) — запрос от программы к сервису. Нижняя (зелёная) — ответ: JSON от сервиса обратно к программе. Внизу подпись: всё это происходит без ручного скачивания — программа сама получает актуальные данные.

Разложим цикл на шаги на примере «узнать курс доллара»:

  1. Программа формирует запрос. Она обращается по специальному адресу (URL) сервиса и говорит, что хочет: «дай курс USD к KZT». Этот адрес и есть «точка входа» API.
  2. Запрос уходит к сервису по сети. Программа отправляет его так же, как браузер отправляет запрос, когда ты открываешь сайт.
  3. Сервис обрабатывает запрос. Он понимает, о чём его спросили, и берёт нужные данные из своей базы.
  4. Сервис возвращает ответ. Чаще всего в формате JSON — компактном и понятном для программ.
  5. Программа разбирает ответ. Она читает JSON, достаёт нужное значение по ключу и использует его — показывает на экране, сохраняет, считает.

Как выглядит ответ JSON на такой запрос — упрощённо:

{
  "base": "USD",
  "to": "KZT",
  "rate": 478.52,
  "date": "2026-06-26"
}

Программе достаточно взять значение по ключу rate — и она знает курс. Никакого ручного скачивания, никакого копирования из таблицы: данные пришли структурированными, бери и используй.

Теперь — не абстрактный пример, а реальный API портала. У data.egov.kz тоже есть API, и самое приятное — простой GET-запрос можно отправить прямо из адресной строки браузера, без единой строчки кода. Адрес запроса выглядит так:

https://data.egov.kz/api/v4/<имя_набора>/<версия>?apiKey=ВАШ_КЛЮЧ

Разберём этот URL по частям, как мы разбираем всё в этом курсе:

  • https://data.egov.kz/api/v4/ — точка входа API портала, версия v4;
  • <имя_набора> — идентификатор конкретного набора данных (его берёшь из карточки набора на портале);
  • <версия> — версия набора;
  • ?apiKey=ВАШ_КЛЮЧ — твой личный ключ доступа, который выдаётся после регистрации на портале. Без ключа сервер запрос не примет — это нормально, так порталы понимают, кто и сколько запросов шлёт.

Вставил такой адрес в браузер, нажал Enter — и сервер в ответ отдаёт данные в формате JSON, например так:

[
  {"region": "Алматы", "year": 2024, "population": 2228000},
  {"region": "Астана", "year": 2024, "population": 1430000}
]

Читается это ровно как JSON из раздела 4.3: массив [ ] записей, в каждой { } пары «ключ–значение» — регион, год, население. Разница с файлом только в том, что этот JSON тебе не пришлось скачивать руками и открывать — он пришёл прямо в ответ на запрос. Захотел свежие цифры завтра — отправил тот же запрос ещё раз, и данные обновились сами. Вот в этом и есть главное преимущество API над файлом: данные приходят свежими и автоматически.

Важная мысль о «запросе» и «ответе». Это тот самый ответ, который ты должен уметь дать на экзамене и в задании: программа отправляет сервису запрос, сервис возвращает ответ с данными — обычно в формате JSON — и программа сразу использует свежие данные, без ручного скачивания файла. Запомни пару «запрос → ответ» как сердце работы любого API. Большинство таких API построены по стилю REST — это распространённый подход к организации запросов и ответов, который ты подробно изучишь в темах автоматизации [4; 6].

Где здесь открытые данные. Многие порталы открытых данных, включая data.egov.kz, дают доступ к наборам двумя способами: файлом и через API. То есть один и тот же набор можно либо скачать (XLSX/JSON/XML), либо запрашивать программно через API и получать JSON. Что выбрать — зависит от задачи, об этом следующий подраздел.

4.6. Файл или API: когда что выбирать

Это практический вопрос, который встаёт в каждом проекте. Ответ зависит от двух вещей: как часто меняются данные и нужна ли автоматика.

Скачай файл один раз, если:

  • данные меняются редко или вообще статичны (население за прошлые годы, справочник регионов);
  • тебе нужен разовый анализ или учебный дашборд;
  • ты хочешь просто посмотреть и обработать данные вручную.

Используй API, если:

  • данные меняются часто (курс валют, погода, расписание);
  • программа должна обновлять данные автоматически, без участия человека;
  • ты строишь сервис, который постоянно опирается на чужие данные.

Бытовой пример, чтобы засело. Список твоих одноклассников меняется раз в год — его проще держать файлом и обновлять вручную. А время прибытия автобуса меняется каждую минуту — его нет смысла «скачивать», его надо запрашивать через API ровно в тот момент, когда ты смотришь на табло. Тот же принцип в коде: редкое и статичное — файлом, частое и живое — через API. И честная оговорка: для учебного проекта чаще всего хватает файла — не надо усложнять и городить API там, где данные всё равно за прошлый год. API включай тогда, когда файл реально устаревает быстрее, чем ты успеваешь его обновлять.

4.7. Открытые против персональных: где проходит граница

Это самый важный для ответственности подраздел темы. Открытые данные публичны — но «публичны» не значит «публиковать можно что угодно». Есть жёсткая граница, и переходить её нельзя даже из лучших побуждений.

Открытые данные — обезличенные и агрегированные: «в Алматы 2 млн жителей», «в Туркестанской области 800 школ». По ним нельзя опознать человека. Их можно и нужно публиковать.

Персональные данные — сведения, по которым можно прямо или косвенно опознать конкретного человека: ФИО, ИИН, номер телефона, адрес, данные документов. Публиковать их как открытые нельзя. Это запрещает Закон РК «О персональных данных и их защите» от 21.05.2013 № 94-V [3], а сферу информатизации в целом регулирует Закон РК «Об информатизации» № 418-V [2].

Сравни на примере одной темы — «школы региона»:

Можно публиковать как открытые данныеНельзя — это персональные данные
число школ по областямсписок учеников с ФИО и адресами
количество учеников всеготелефоны родителей
статистика по успеваемости по регионуоценки конкретного ребёнка с его именем

Видишь принцип: как только данные становятся про конкретного человека, а не про общую массу, они превращаются в персональные — и публиковать их запрещено. Открытые данные всегда обобщены до уровня, где отдельный человек «растворяется» в статистике и его нельзя выцепить.

Что это значит для тебя на практике. Во-первых, из открытых источников бери только обезличенные данные — на data.egov.kz они такие и есть, за тебя об этом уже позаботились. Во-вторых, если ты сам работаешь с данными, где есть персональные сведения (в учебном задании или на работе), не публикуй их и не загружай в публичные сервисы — сначала обезличь: убери ФИО, ИИН, телефоны, замени имена на «Клиент 1», «Ученик 2». Отдельно предостерегу: не тащи таблицы с реальными ФИО и ИИН в чужие онлайн-сервисы «просто посмотреть» — как только данные ушли на чужой сервер, ты их уже не контролируешь. Этот навык обезличивания и принцип дата-минимизации ты встречал в теме 01 и продолжишь применять при работе с ИИ и данными дальше [3].

5. Разобранные примеры и мини-кейсы

Пример 1 — выдуманные данные против настоящих. Студент строил дашборд «население городов» и вписал цифры «на глаз». Преподаватель попросил реальные данные.
Разбор: выдуманные числа делают любые выводы недостоверными — на них нельзя ничего утверждать. Правильный путь: зайти на data.egov.kz, найти набор по населению, скачать его (XLSX или JSON — портал отдаёт эти форматы) и перестроить дашборд на настоящих числах, указав ссылку на источник. Если бы данные обновлялись постоянно — стоило бы брать их через API. Это и есть разница между «проектом для вида» и серьёзной работой.

Пример 2 — выбор формата под задачу. Тебе нужно (а) быстро глянуть набор глазами и (б) обработать тот же набор программой на Python.
Разбор: для (а) удобнее XLSX — открыл в Excel и посмотрел; для (б) нужен формат, который легко читает код, — JSON (или CSV, если данные плоские). Один и тот же набор может понадобиться в разных форматах под разные шаги работы. Поэтому, выбирая формат при скачивании с портала, сразу думай: это «для глаз» или «для кода».

Пример 3 — файл или API. Тебе нужно показывать в приложении актуальный курс валют.
Разбор: скачивать файл с курсом бессмысленно — он устареет через минуту. Здесь нужен API: программа шлёт запрос к сервису и получает ответ JSON с текущим курсом, обновляя его автоматически. А вот если бы тебе нужна была история курса за прошлый год для анализа — это статичные данные, и их разумнее один раз скачать файлом.

Пример 4 — читаем реальный API-запрос. Тебе дали строку https://data.egov.kz/api/v4/<имя_набора>/<версия>?apiKey=ВАШ_КЛЮЧ и спросили, что это.
Разбор: разложи по частям, как всё в этом курсе. https://data.egov.kz/api/v4/ — точка входа API портала (версия v4); <имя_набора>/<версия> — какой именно набор и какой его версии мы просим; ?apiKey=… — личный ключ доступа, без него сервер запрос не примет. Вставляешь такой URL в адресную строку браузера, жмёшь Enter — и сервер возвращает данные в JSON (массив записей с ключами вроде region, year, population). Никакого кода не нужно, чтобы увидеть, как работает API, — достаточно браузера.

Пример 5 — граница приватности. Чиновник предлагает «для прозрачности» опубликовать как открытые данные список всех получателей пособия с ФИО и ИИН.
Разбор: это нарушение Закона РК № 94-V [3]. Список конкретных людей с идентификаторами — персональные данные, их публиковать нельзя. Открытыми могут быть только обезличенные показатели: «сколько человек получили пособие по области», «средний размер выплаты». Прозрачность достигается обобщёнными цифрами, а не раскрытием личностей.

6. Частые ошибки и как их избежать

  • Ошибка: использовать выдуманные данные «для вида». Почему: выводы становятся недостоверными, проект — несерьёзным, а первый же вопрос «откуда?» его рушит. Как правильно: бери реальные наборы с data.egov.kz и указывай ссылку на источник.
  • Ошибка: считать, что «открытые данные» означают «можно публиковать что угодно о людях». Почему: персональные данные защищены законом и не могут быть открытыми. Как правильно: публикуй и используй только обезличенные, агрегированные данные.
  • Ошибка: думать, что data.egov.kz отдаёт CSV, и искать кнопку CSV. Почему: портал отдаёт XLSX, JSON и XML; CSV — это формат других источников. Как правильно: с портала бери JSON (для кода) или XLSX (для просмотра); CSV конвертируешь сам, если нужен именно он.
  • Ошибка: скачивать XLSX, чтобы потом обрабатывать его кодом. Почему: XLSX неудобен для автоматической обработки — оформление и объединённые ячейки мешают. Как правильно: для кода бери JSON или CSV, XLSX — только для просмотра.
  • Ошибка: путать запрос и ответ при описании API. Почему: теряется понимание, как идёт обмен. Как правильно: запомни направление: программа шлёт запрос → сервис возвращает ответ (обычно JSON).
  • Ошибка: взять набор, но не записать его реквизиты и ссылку. Почему: нельзя доказать достоверность, а потом сам не найдёшь источник среди тысяч наборов. Как правильно: сразу фиксируй название, формат, период и прямую ссылку.

7. Памятка: «От задачи до данных»

Короткий алгоритм работы с открытыми данными и API:
  1. Сформулируй, какие данные нужны — тема, период, какие колонки.
  2. Реши, файл или API — данные статичные? файлом. Меняются часто? через API.
  3. Найди набор на data.egov.kz — поиск по теме, открой карточку, проверь период и формат.
  4. Выбери формат — для кода JSON/CSV, для просмотра XLSX (помни: портал отдаёт XLSX/JSON/XML).
  5. Скачай и зафиксируй реквизиты — название, формат, период, ссылку.
  6. Проверь приватность — в данных не должно быть персональных сведений; если есть — обезличь.
  7. Используй — построй дашборд, сделай выводы, при необходимости настрой получение через API.

8. Краткие итоги

  • Открытые данные — обезличенные наборы, опубликованные свободно и бесплатно для повторного использования; бери их вместо выдуманных цифр.
  • Главный портал открытых данных Казахстана — data.egov.kz; ищи набор по теме, читай карточку, фиксируй реквизиты и ссылку.
  • Портал отдаёт наборы в XLSX, JSON и XML; CSV — распространённый формат у других источников. Для кода бери JSON/CSV, для просмотра — XLSX.
  • API — способ программно получать данные у чужого сервиса: программа шлёт запрос → сервис возвращает ответ, обычно в формате JSON. У data.egov.kz есть API вида .../api/v4/<набор>/<версия>?apiKey=....
  • Файл скачивай для статичных данных, API используй для часто меняющихся.
  • Открытые данные всегда обезличены; персональные данные публиковать как открытые нельзя — это запрещено Законом РК № 94-V.

10. Глоссарий

  • API (Application Programming Interface) — программный интерфейс: способ одной программы запрашивать данные или функции у другой.
  • CSV — текстовый табличный формат, где значения в строке разделены запятыми; удобен и для кода, и для Excel.
  • JSON — текстовый формат записи данных в виде пар «ключ–значение»; основной формат ответов API.
  • XLSX — формат таблиц Microsoft Excel; удобен для просмотра человеком, неудобен для кода.
  • apiKey — личный ключ доступа к API портала, выдаётся после регистрации; без него запрос не принимается.
  • Запрос (request) — обращение программы к API с просьбой предоставить данные или выполнить действие.
  • Набор данных (датасет) — структурированная коллекция данных по одной теме.
  • Обезличивание — удаление из данных признаков, по которым можно опознать конкретного человека.
  • Обновление данных — изменение содержимого набора во времени; влияет на выбор между файлом и API.
  • Ответ (response) — данные, которые API возвращает в ответ на запрос, обычно в формате JSON.
  • Открытые данные — обезличенные наборы, опубликованные свободно и бесплатно для повторного использования.
  • Персональные данные — сведения, по которым можно прямо или косвенно опознать конкретного человека.
  • Портал открытых данных — сайт, где публикуются и каталогизируются наборы открытых данных (в РК — data.egov.kz).
  • REST — распространённый стиль организации API на основе запросов и ответов по сети.
