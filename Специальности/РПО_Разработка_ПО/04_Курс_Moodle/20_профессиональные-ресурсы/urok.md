# Профессиональные ресурсы разработчика: Stack Overflow, MDN, GitHub

Разработчик не держит всё в голове — он умеет быстро найти ответ в правильном месте. Документация подскажет, как работает функция; Stack Overflow — как решали похожую проблему другие; GitHub — где живёт код и как над ним работают командой. Знать эти ресурсы и пользоваться ими грамотно — профессиональный навык не меньше, чем знание языка.

## Чему ты научишься

- понимать назначение ключевых ресурсов разработчика;
- правильно искать ответ и оценивать его;
- ориентироваться в GitHub как месте хранения кода.

## Три опоры разработчика

| Ресурс | Для чего | Когда идти |
|---|---|---|
| Документация (MDN, docs.python.org) | как что-то работает официально | первоисточник, основа |
| Stack Overflow | как другие решали конкретную проблему | застрял с ошибкой |
| GitHub | хранение кода, версии, командная работа | свой и чужой код |

## Документация — первоисточник

Официальная документация (MDN для веба, docs.python.org для Python) — самый надёжный источник. Там описано, как работает функция, какие параметры принимает, примеры. Начинай с неё.

## Stack Overflow — опыт сообщества

Q&A-площадка: кто-то задал вопрос, другие ответили; лучшие ответы поднимаются голосами. Полезно, но:

- смотри **дату** и **число голосов**;
- проверяй, подходит ли ответ к твоей версии;
- понимай код, прежде чем копировать.

## GitHub — дом кода

**GitHub** — платформа для хранения кода с системой версий (git). Здесь: репозитории, история изменений, совместная работа, открытые проекты (open source). Твоё портфолио разработчика тоже живёт на GitHub.

### Мини-кейс
Студент скопировал ответ со Stack Overflow с 2 голосами от 2013 года — не сработало. Рядом был ответ 2024 года с 300 голосами. Следующий шаг: выбирать свежие ответы с высоким рейтингом и сверять с документацией.

## Частые ошибки

**Ошибка:** копировать первый попавшийся ответ, не читая обсуждение.
**Почему:** ответ может быть устаревшим или не для твоего случая.
**Как правильно:** смотреть дату/голоса, читать комментарии, сверять с документацией.

**Ошибка:** хранить код только на своём ПК.
**Почему:** поломка = потеря; нет истории и совместной работы.
**Как правильно:** держать код в репозитории на GitHub.

## Итог: что теперь делать

- Начинай с документации — это первоисточник.
- На Stack Overflow выбирай свежие ответы с рейтингом и понимай код.
- Храни код на GitHub: версии, история, портфолио.
- Любой найденный код проверяй, прежде чем использовать.
