Задание к занятию 01. Мини-аудит ответственного использования ИИ

Стандарт задания: вариант — по номеру в журнале (один кейс на ~5 человек), внутри варианта — обязательная часть и продвинутая (на доп. баллы). Это защищает от списывания и даёт рост сильным студентам.
№ 1–5 → Вариант 1 · № 6–10 → Вариант 2 · № 11–15 → Вариант 3 · № 16–20 → Вариант 4 · № 21–25 → Вариант 5 · № 26–30 → Вариант 6.

Обязательная часть (до 85 баллов) — для любого варианта

  1. Разбери действия команды: по каждому пункту кейса — ответственное использование ИИ или перекладывание ответственности (с обоснованием).
  2. Риски: минимум 3 конкретных (безопасность, приватность, достоверность) с привязкой к пунктам.
  3. Право: назови нарушенный закон РК и пункт кейса.
  4. Исправь процесс: как переделать работу с ИИ правильно.
Продвинутая часть (+15 баллов): разработай мини-политику ответственного ИИ для команды — 6–8 правил по разделам «Код», «Данные/приватность», «Достоверность», «Ответственность». Защищается устно.

Кейсы по вариантам

Вариант 1. Учебный веб-сервис «Запись на консультацию». 1) ИИ-функцию аутентификации вставили не читая; 2) загрузили в чат-бота ФИО, ИИН и телефоны знакомых «для теста»; 3) описание проекта целиком от ИИ без проверки фактов; 4) архитектуру от ИИ приняли без обсуждения; 5) участник нашёл в сгенерированном SQL уязвимость к инъекции и исправил.

Вариант 2. Мобильное приложение «Доставка еды». 1) модуль оплаты от ИИ подключили к реальной платёжке без ревью; 2) для теста геолокации загрузили в ИИ адреса и телефоны курьеров; 3) текст уведомления от ИИ содержал неверную формулировку о возврате; 4) токены по совету ИИ хранят в открытом виде в коде; 5) участница нашла статанализом утечку ключа API.

Вариант 3. Чат-бот поддержки колледжа. 1) бота обучили на реальной переписке студентов без согласия; 2) ответы публиковали без проверки — один с выдуманным сроком подачи документов; 3) код интеграции с базой от ИИ не читали; 4) персональные обращения пересылали в публичный ИИ-сервис; 5) разработчик добавил проверку ответов модератором.

Вариант 4. Пет-проект «Анализатор резюме». 1) парсер от ИИ подключили, не разобравшись; 2) для теста загрузили реальные резюме с ФИО и ИИН; 3) оценку кандидатов доверили ИИ без проверки на предвзятость; 4) README с выдуманной точностью от ИИ; 5) участник заметил занижение оценок по полу/возрасту и убрал эти признаки.

Вариант 5. Интернет-магазин (учебный). 1) SQL от ИИ вставили без защиты от инъекций; 2) отзывы сгенерировал ИИ и выдали за настоящие; 3) базу клиентов выгрузили в ИИ «придумать акции»; 4) код корзины приняли без тестов; 5) участница автотестами нашла ошибку округления цены.

Вариант 6. Фриланс-заказ «CRM для клиники». 1) реальные данные пациентов загрузили в ИИ для демо; 2) модуль записи к врачу от ИИ внедрили без ревью; 3) текст согласия на обработку данных от ИИ не соответствует закону РК; 4) согласились на SMS-напоминания без согласия пациентов; 5) разработчик настоял на обезличивании данных и переделал датасет.

СРС (самостоятельная работа, 2 ч)

Тема СРС: Эссе: цифровые технологии в работе разработчика.
Задание: Напиши эссе (250–350 слов) о роли цифровых технологий в работе разработчика ПО. Используй обезличенные данные; укажи, если применял ИИ.
Критерий приёмки: раскрыты 3–4 технологии и их роль, есть вывод, текст самостоятельный.
