# Упражнения. Занятие 27. Промпт-инжиниринг для кодовых задач

> Формат — составление и улучшение промптов. Критерии приёмки под каждым заданием.

## Упражнение 1. Разбери промпт на части
Возьми промпт «Ты Python-разработчик. Напиши функцию проверки e-mail: строка → True/False, только стандартная библиотека, дай код и пояснение». Найди роль, задачу, вход/выход, ограничение, формат.

**Критерий приёмки:** все 5 частей выделены верно.

## Упражнение 2. Улучши плохой промпт
Перепиши «помоги с кодом для парсинга» в структурированный промпт с примером вход→выход.

**Критерий приёмки:** есть язык, задача, вход/выход, ограничение и пример.

## Упражнение 3. Добавь пример (few-shot)
К задаче «привести строку к заголовочному регистру» добавь 2 примера вход→выход.

**Критерий приёмки:** 2 корректных примера (например `hello world` → `Hello World`).

## Упражнение 4. Итерация
Первый ответ ИИ не обработал пустую строку. Как уточнить промпт, чтобы это исправить?

**Критерий приёмки:** добавить требование/пример обработки пустого ввода и крайних случаев.

## Упражнение 5. Оцени два промпта
Сравни «сделай сортировку» и «Напиши на Python функцию сортировки списка чисел по убыванию, без сторонних библиотек; пример [3,1,2]→[3,2,1]». Какой лучше и почему?

**Критерий приёмки:** второй; конкретен, есть язык, направление сортировки, ограничение и пример.

## Расширенная практика (на 2 пары)

> Эти задания рассчитаны на 4 часа (2 пары) практической работы за компьютером с ИИ-ассистентом. Работаешь с любым доступным ИИ (Copilot, ChatGPT, Gemini, Claude и т. п.) на тестовых/обезличенных данных. Реальный код проверяешь сам: запускай и тестируй, не верь ответу ИИ на слово.

### Пара 1. «Плохой» против «хорошего» промпта: сравнение и разбор

**Цель:** на собственном опыте увидеть, какие приёмы превращают расплывчатый запрос в точный, и научиться объяснять это словами.

**Что делаешь.** Возьми 4 типовые кодовые задачи (можно эти, можно свои похожего уровня):

1. функция: вернуть список чётных чисел из списка чисел;
2. функция: посчитать, сколько раз слово встречается в строке (без учёта регистра);
3. функция: проверить, является ли строка палиндромом (игнорируя пробелы и регистр);
4. функция: из строки вида `имя=значение;имя=значение` собрать словарь (пример: `a=1;b=2` → `{'a': '1', 'b': '2'}`).

Для каждой задачи:

1. Напиши **«плохой» промпт** — короткий и общий (например: «сделай функцию для чётных чисел»). Отправь ИИ, сохрани ответ.
2. Напиши **«хороший» промпт** — по схеме урока: роль + задача + вход/выход + ограничения + формат + пример вход→выход. Отправь ИИ, сохрани ответ.
3. **Запусти оба варианта кода** на 2–3 своих входных данных, включая один крайний случай (пустой список/строка). Отметь, какой код заработал сразу, а какой пришлось дорабатывать.
4. В таблице сравни результаты и **объясни своими словами**, какой приём сработал: конкретика, пример (few-shot), пошаговость (chain-of-thought — «сначала план, потом код»), итерация.

**Образец таблицы сравнения (заполни по каждой из 4 задач):**

| Задача | «Плохой» промпт: что получил | «Хороший» промпт: что получил | Какой приём решил исход |
|---|---|---|---|
| Чётные числа | вернул код без обработки пустого списка, тип входа угадан неверно | рабочий код с первого раза, есть проверка пустого входа | конкретика входа/выхода + пример `[1,2,3,4]`→`[2,4]` |

**Дополнительно (мини-эксперимент с пошаговостью).** Для одной из 4 задач отправь ИИ ещё один промпт с явной пошаговостью: «Сначала опиши план решения по шагам, потом дай код». Сравни с обычным «хорошим» промптом: стало ли решение надёжнее, появились ли крайние случаи в плане.

**Критерии приёмки (Пара 1):**

- по всем 4 задачам есть пара «плохой/хороший» промпт и сохранённые ответы ИИ;
- оба варианта кода **запущены** хотя бы на одном крайнем случае, результат записан;
- таблица сравнения заполнена по всем 4 задачам;
- для каждой задачи **своими словами** названо, какой приём (конкретика / few-shot / пошаговость / итерация) дал лучший результат и почему;
- выполнен мини-эксперимент с пошаговостью хотя бы для одной задачи;
- **устная защита:** на 1–2 задачах ты можешь, не читая с листа, объяснить разницу между промптами и почему хороший дал лучший код.

### Пара 2. Личная библиотека промптов под реальные задачи

**Цель:** собрать рабочий набор промптов, который ты сможешь использовать в учёбе и на практике, и измерить, насколько он эффективен.

**Что делаешь.** Составь **личную библиотеку из 5–7 промптов** — по одному на каждый типовой сценарий разработчика:

1. **Генерация функции** — например: «Ты Python-разработчик. Напиши функцию `<имя>`: вход `<...>`, выход `<...>`; ограничения `<...>`; дай код и 2 примера вход→выход».
2. **Объяснение чужого кода** — «Объясни по шагам, что делает этот код и какие у него крайние случаи: `<вставить код>`».
3. **Рефакторинг** — «Перепиши этот код понятнее и короче, не меняя поведение; объясни, что улучшил: `<код>`».
4. **Генерация тестов** — «Предложи тесты для этой функции: обычный, крайний и некорректный ввод; формат — pytest: `<код>`».
5. **Отладка** — «Вот код и текст ошибки. Объясни причину и предложи исправление, не маскируя баг: `<код>` / `<traceback>`».
6–7. (по желанию) свои сценарии: генерация регулярного выражения, написание docstring, перевод псевдокода в код и т. п.

Для каждого промпта:

1. Применяй его к **реальной мелкой задаче** (тестовые/обезличенные данные, без секретов и личных данных).
2. **Запусти результат** и доводи до рабочего состояния, уточняя промпт.
3. Зафиксируй **число итераций до рабочего результата** (сколько раз пришлось переписать запрос).
4. Запиши, **как именно ты применял ИИ** (для генерации / объяснения / проверки) — это требование AI-устойчивости.

**Образец карточки промпта (сделай по каждому из 5–7):**

```
Сценарий: генерация тестов
Промпт: «Предложи тесты pytest для функции average(numbers):
         обычный список, один элемент, пустой список, нечисловой ввод. Дай код.»
Задача, на которой проверял: функция среднего арифметического
Итераций до рабочего результата: 2 (в 1-й раз не было теста на пустой список — уточнил)
Как применял ИИ: сгенерировал черновик тестов, проверил и дописал assert сам
Чем хорош промпт: задаёт три вида случаев и формат (pytest) — ответ почти готов
```

**Сводная таблица библиотеки:**

| № | Сценарий | Итераций до рабочего результата | Чем хорош промпт |
|---|---|---|---|
| 1 | Генерация функции | 1 | конкретный вход/выход + пример |
| 2 | Объяснение кода | 1 | просьба разобрать по шагам и назвать крайние случаи |
| … | … | … | … |

**Критерии приёмки (Пара 2):**

- в библиотеке **5–7 промптов**, покрыты все 5 обязательных сценариев (генерация функции, объяснение, рефакторинг, тесты, отладка);
- каждый промпт **применён к реальной задаче** и доведён до рабочего кода (код запускался);
- по каждому промпту зафиксировано **число итераций** до рабочего результата;
- по каждому промпту указано, **как применялся ИИ** (генерация / объяснение / проверка);
- заполнена сводная таблица;
- **устная защита:** ты можешь показать 2–3 промпта из своей библиотеки и объяснить, почему они сформулированы именно так и что бы изменил при большем числе итераций.

> **AI-устойчивость.** Защита устная: преподаватель может попросить переформулировать любой промпт «на ходу» под новую задачу и объяснить выбор приёмов. Готовые промпты, скопированные без понимания, на устной защите видны сразу.
